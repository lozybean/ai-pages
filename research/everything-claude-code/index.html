调研口径

https://github.com/affaan-m/everything-claude-code 当前会解析到实际仓库 affaan-m/ECC。分析基于 main 快照 71d22d0a77b7e0684f4e51cba03749b788993cdb，并交叉阅读 README、.codex/、skills、rules、hooks、install manifests、MCP policy、security guide 与测试/CI 脚本。

README badge 中的 stars/forks 等营销数字不作为判断依据。真正影响使用建议的是文件结构、安装边界、Codex 适配方式、hook/MCP 的执行风险，以及哪些工作流能被迁移到当前 Codex。

核心判断

ECC 最值得借鉴的不是“装一套 Claude Code 配置”，而是把 agent 工作拆成可复用的 workflow primitives：skills、rules、verification、security boundary、context strategy、MCP selection 和 multi-agent handoff。

推荐吸收

把 ECC 当成工作流模式库。优先吸收 tdd-workflow、verification-loop、security-review、strategic-compact、deep-research、语言/framework rules、MCP 最小化原则、以及“研究 → 计划 → 实现 → review → 验证”的阶段化交付习惯。

不建议照搬

不要把 full profile、全量 rules、全量 MCP、全局 git hooks 和 yolo 形态直接装到日常机器。Codex 当前没有 Claude Code hook parity，ECC 的 Codex plugin 路径也被仓库自身标注为 fragile；更稳的是项目级选择性同步、dry-run、最小 MCP 和人工审批。

仓库形态：从 Claude Code 配置库演化成跨 harness 操作系统

仓库 README 已把 ECC 定义成跨 Codex、Claude Code、Cursor、OpenCode、Gemini、Zed、GitHub Copilot 等 harness 的 operator system。package.json 也把 .agents/、.codex/、.cursor/、.opencode/、rules/、hooks/、skills/、mcp-configs/ 和安装脚本都列为发布面。

层 仓库中的证据 工程含义
Skills skills/*/SKILL.md 共 271 个，.agents/skills 也提供 Codex/OpenAI 元数据 这是最值得复用的单元。它比“把所有原则塞进 AGENTS.md”更可控，适合按任务触发。
Rules rules/common 加 TypeScript、Python、Go、Java、React、Rust 等语言包 适合作为项目规范素材，但要按团队风格裁剪。全量加载会带来上下文和冲突成本。
Agents 根 AGENTS.md 列出 planner、reviewer、security、build resolver 等角色 适合抽象成 Codex multi-agent 角色，但父 agent 必须带目标背景，不应只丢一个模糊查询。
Commands 92 个 legacy command shims；根说明把 skills/ 定为 canonical workflow surface 日常使用应优先 skills-first。Slash command 更像入口别名，不是长期维护中心。
Hooks hooks/ 和 scripts/hooks/ 覆盖格式化、质量门禁、上下文提醒、成本监控等 Claude Code/Cursor/OpenCode 更可用；Codex 不能等价迁移，应改成显式验证命令、git hook 或人工 checklist。
MCP MCP policy 把默认 connector 收缩为 chrome-devtools，其他服务改为 opt-in 这是正确方向：默认少开连接器，能用 CLI/API + skill 解决的，不要常驻 MCP。

Codex 推荐用法

1. 把 ECC 当作候选素材库，不要直接覆盖本机全局配置

ECC 的 Codex 同步脚本会备份 ~/.codex/config.toml 和 ~/.codex/AGENTS.md，用 marker 合并 AGENTS 内容，并把 baseline config 以 add-only 方式合入。这个设计方向是对的，但它仍会触及用户全局目录、prompt 文件和 git hooks。推荐先在临时环境或测试账号中跑 dry-run，确认差异后再决定是否吸收。

git clone https://github.com/affaan-m/ECC.git
cd ECC
npm install
bash scripts/sync-ecc-to-codex.sh --dry-run

2. Codex 插件路径先当 discovery，不当主安装路径

.codex-plugin/README.md 自己说明 Codex repo marketplace 安装目前存在 runtime skill loading 不可靠的问题，并建议使用 manual sync。实际推荐是：需要研究 ECC 时可看 plugin manifest 和 marketplace；真正落到日常 Codex，则优先用项目内 AGENTS.md、少量 .agents/skills、项目级 .codex/config.toml 和明确的验证脚本。

3. 复制 config 思路，而不是复制每一行 config

.codex/config.toml 中比较稳的思路是：approval_policy = "on-request"、sandbox_mode = "workspace-write"、严格 profile 使用 read-only、多 agent 角色只做一层浅委派。需要谨慎的是模型名、默认 MCP 集、通知命令和 yolo profile；这些都应按本机 Codex 版本和当前项目风险复核。

4. Codex 没有 hook parity，安全边界应落在 sandbox、approval 和验证命令

ECC 的 Codex supplement 明确列出差异：Claude Code 有 hook 事件，Codex 侧主要靠 AGENTS.md、skills、config.toml、MCP 与 sandbox。迁移到 Codex 时，不要把 Claude hook 中的自动格式化、拦截、记忆、质量门禁理解为“已经自动生效”。更实际的做法是把它们变成：

  • 提交前显式运行的 test/lint/typecheck/security 命令；
  • 项目级 CI 或 repo-local git hooks；
  • Codex 中每次写文件/提交前必须执行的 verification checklist；
  • 对 network、off-repo write、secret path、deploy、push 的审批边界。

5. MCP 默认少开，任务需要时再启用

README 早期内容还保留 GitHub、Context7、Exa、Memory、Playwright、Sequential Thinking 等默认集合的说法；但 MCP policy 和 merge-mcp-config.js 已经把当前默认集收缩为 chrome-devtools。这反而是最值得采用的原则：在 Codex 中优先使用内建搜索、CLI、一次性 API 调用和 task-scoped skill；只有浏览器调试这类需要持久交互会话的场景，才默认考虑 MCP。

日常工作推荐用法

场景 推荐用法 从 ECC 借鉴的具体点
新功能 / bugfix 中大型任务先写 plan，再进入 RED → GREEN → refactor → evidence report。小任务可直接做，但仍要跑相关验证。 tdd-workflow 把计划视为不可信输入，要求先转成可测试保证，并保留 RED/GREEN 证据。
提交前验证 把 build、typecheck、lint、test、security scan、diff review 固化成项目脚本或 checklist。 verification-loop 的阶段化质量门禁可迁移到 Codex prompt、CI 或 repo-local scripts。
安全敏感改动 对 auth、输入、上传、API、secrets、支付、第三方集成强制安全 review；不要让模型绕过审批边界。 security-review、security guide 和 SECURITY.md 都把 secrets、输入验证、权限、供应链当成一等风险面。
长会话 在“探索 → 计划”“计划 → 实现”“debug → 新任务”这些边界 compact；不要在实现一半时 compact。 strategic-compact 的价值不是自动压缩，而是按任务阶段清理上下文。
研究和报告 拆成 3-5 个子问题，优先读取 primary source，区分事实与推断，长报告落成文件。 deep-research 的质量规则很适合 Codex：每个关键 claim 要有来源，单一来源要标为未充分验证。
多 agent / 并行 只有任务彼此独立时才并行；并行修改代码必须用 worktree 或明确文件边界；子 agent 输出要被父 agent 复核。 longform guide 的 iterative retrieval pattern 和 sequential phases 比“开很多终端”更可靠。
团队规范沉淀 把稳定规则放到项目 AGENTS.md 或 docs；把任务型流程放进 skill；把临时探索结论写成报告或记忆摘要。 ECC 的 skills-first 和 rules 分层能减少 AGENTS.md 膨胀，但必须避免重复上下文。

推荐采用路径

阶段一：只读评估

先不要安装。把 ECC 当成参考仓库读三类文件：skills/*/SKILL.md、rules/common 与 docs/MCP-CONNECTOR-POLICY.md。如果目标是 Codex，重点看 .codex/AGENTS.md、.codex/config.toml、.codex/agents/*.toml 和 scripts/sync-ecc-to-codex.sh。

阶段二：选择性 dry-run

在 ECC 仓库安装依赖后，用 consult/plan 先看安装计划。裸 clone 下 install-plan.js 会因为缺少依赖而失败，这是正常的源码使用前提，不应跳过。

npm install
node scripts/consult.js "security review" --target codex --json
node scripts/install-plan.js --profile minimal --target codex --json
node scripts/install-plan.js --profile minimal --target codex --with skill:security-review --json

阶段三：迁移少量高价值资产

最值得先迁移的不是 full profile，而是少量稳定资产：

  • security-review：作为 auth/API/secrets 变更的强制 checklist；
  • verification-loop：作为提交前报告模板；
  • strategic-compact：作为长任务上下文策略；
  • tdd-workflow：中大型行为变更时使用，但要按当前项目实际测试体系调整；
  • deep-research：研究任务使用，但工具名和搜索 API 要按当前环境复核。

阶段四：项目级固化

把筛选后的规则落到项目自身的 AGENTS.md、docs/、CI、Makefile/npm scripts 或 repo-local hooks，而不是依赖用户全局状态。对团队协作而言，项目级可复现比个人全局增强更重要。

风险边界与反模式

不要 stack install methods

README 明确警告：Claude plugin、manual installer、full profile 叠加会制造重复 skills、重复 runtime 行为和难排查的状态。Codex 也应遵循同样原则：一个环境只保留一条清晰安装路径。

谨慎使用全局 git hooks

scripts/codex/install-global-git-hooks.sh 会设置 git config --global core.hooksPath。这会影响所有仓库。除非你明确希望全局接管 hooks，否则更推荐把相同逻辑改成项目级 pre-commit、CI 或手动验证命令。

不要把 yolo profile 当日常默认

ECC 的 security guide 强调审批边界：网络、unsandboxed shell、secret path、repo 外写入、部署和 workflow dispatch 都不应由模型最终拍板。Codex 日常默认应保持 on-request 和 workspace-write，高风险任务用更窄 sandbox。

不要把 skills / hooks / MCP 当无害文本

ECC 自己的 security guide 与 SECURITY.md 都把 agent 配置、hooks、MCP、skills、prompt 文件视为供应链面。安装别人给的 skill 前，应像审 npm package 一样审它是否读 secrets、发网络请求、改权限或引入隐藏指令。

注意仓库内部文档漂移

同一快照里，.codex/AGENTS.md 和 .codex/config.toml 仍描述多个默认 MCP，而 MCP policy 与 merge 脚本已经收缩为一个默认 connector。遇到冲突时，优先看当前脚本和 policy，再看 README 叙述。
