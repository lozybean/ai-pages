Runtime API、SSE 与远程控制路径

CodeWhale 的远程能力不是内置公网 tunnel，而是 local-first runtime API 加受控桥接: HTTP/SSE 留在本机或 VPS localhost，手机控制通过 Feishu/Lark 长连接 bridge 转发。

Runtime API

codewhale serve --http 的核心实现位于 crates/tui/src/runtime_api.rs。默认 host 是 127.0.0.1，port 是 7878，workers 默认 2。它启动 task manager、runtime thread manager、automation manager 和 scheduler，然后构造 Axum router。

当前源码的认证默认比部分文档更严格: 如果 CLI token 和 DEEPSEEK_RUNTIME_TOKEN 都没给，且没有显式 --insecure-no-auth，会生成一个本进程临时 bearer token。/health 和 /v1/runtime/info 不在受保护子路由里，其余 /v1/* 路由会经过 token middleware。

路由 能力 认证
/health 返回 deepseek-runtime-api local health。 不受 token middleware 保护。
/v1/runtime/info 返回 bind host、port、auth_required、version。 不受 token middleware 保护。
/v1/threads 创建、列出、查询、更新、resume、fork runtime threads。 默认需要 bearer、x-deepseek-runtime-token 或 query token。
/v1/threads/{id}/turns 启动 turn、steer、interrupt、compact。 默认需要 token。
/v1/approvals/{approval_id} 外部批准或拒绝 pending approval。 默认需要 token。
/v1/tasks / /v1/automations 后台任务、automation、runs、pause/resume。 默认需要 token。
/v1/apps/mcp/* 列出 MCP servers 和 tools。 默认需要 token。

SSE 与线程

Runtime API 提供两种 SSE 场景。/v1/threads/{id}/events 先吐 backlog，再订阅 live event，并用 since_seq 做游标过滤。/v1/stream 是兼容式一次性 stream: 先创建 archived thread，再启动 turn，并把内部 runtime events 映射为兼容事件，直到 turn.completed 后发送 done。

POST /v1/stream
  |
  |-- validate prompt
  |-- choose model, workspace, mode, allow_shell
  |-- create archived runtime thread
  |-- start turn
  |-- replay backlog for that turn
  |-- subscribe live events
  |-- map agent/tool/approval/turn events to SSE
  |-- emit done

这个设计适合本地 UI、桌面壳、手机 bridge 或自动化系统订阅状态，不要求它们直接嵌入 TUI。

App Server

仓库还有 crates/app-server，这是另一个更偏 app protocol 的 HTTP/stdio server。它暴露 /thread、/app、/prompt、/tool、/jobs、/mcp/startup。HTTP 路由默认生成 cwapp_ bearer token，除非显式 --insecure-no-auth，且源码会拒绝 unauthenticated non-loopback bind。

可以把 runtime_api.rs 理解为 TUI runtime 的本地自动化 API，把 app-server 理解为面向独立 app/supervisor 的协议层试验。两者都强调 token 和 loopback，而不是“开一个公网面板”。

ACP 与 MCP

docs/RUNTIME_API.md 把 ACP stdio 描述为保守基线: initialize、session new、prompt、cancel。它有意不通过 ACP 暴露 shell、file-write、checkpoint 或 session loading。完整本地控制走 HTTP API，工具服务集成走 MCP。

MCP 在 runtime API 里有 introspection 路由，主 engine 在执行工具前按需确保 MCP pool。MCP 工具也参与只读/审批/并行判断: 只读工具可自动执行，非只读 MCP 需要审批。

Tencent 远程路径

CodeWhale 的远程 runbook 很明确: Tencent Lighthouse HK 是 always-on host，codewhale-runtime.service 把 runtime API 绑定在 127.0.0.1:7878，codewhale-feishu-bridge.service 通过 Feishu/Lark 长连接接收手机消息，再转发到本机 runtime。

Feishu/Lark mobile app
  -> long-connection bot
  -> codewhale-feishu-bridge systemd service
  -> http://127.0.0.1:7878 codewhale serve --http
  -> /opt/whalebro workspace

CNB mirror 和 deploy templates 用于让 Lighthouse 部署可重复。EdgeOne 被定义为可选 public edge，只应该 front 文档、状态页、未来 webhook 或 demo app，不应该代理 /v1/* runtime endpoints。

Tunnel 判断

不要把 runtime API 当作公网 tunnel 服务

源码和 runbook 都不建议把 http://127.0.0.1:7878 或 /v1/* 暴露到 EdgeOne、Caddy/Nginx 或公网域名。远程控制靠 bridge 和 allowlist，而不是把 token API 开到互联网。

如果要和 cloudflared/ngrok 这类“tunnel”概念对齐，CodeWhale 当前更接近“长连接 bot bridge”: 控制通道由 Feishu/Lark 到桥接服务发起，外部用户不直接连 runtime API。这比公网 tunnel 更符合它 local-first 和 token custody 的安全模型。
