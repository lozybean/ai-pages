Prefix Cache 与上下文经济学

CodeWhale 的 DeepSeek V4 适配重点，不是简单缩短 prompt，而是让稳定前缀可复用、动态尾部可追加、压缩尽量晚发生，并把 cache hit/miss 直接纳入成本视图。

基本模型

README 明确把 DeepSeek V4 prefix caching 作为长 Constitution prompt 可行的前提。源码中的 prefix_cache.rs 把上下文拆成三段: immutable prefix、append-only history、latest user turn。核心目标是减少对前缀字节的改写。

stable prefix
  system prompt + tool names/spec shape + constitution-like rules
        |
append-only history
  prior messages, tool calls, tool results
        |
latest tail
  current user request, transient steer input, new tool results

这个设计和 V4 的 1M context 搭配后，倾向于“先保留长上下文，晚压缩”。对 V4 来说，过早 replacement compaction 会破坏已经热起来的 KV prefix，省 token 的同时可能丢掉 cache 折扣和上下文连续性。

稳定性检测

PrefixStabilityManager 在每轮请求前生成 fingerprint。如果 system prompt hash 或 active tool names 改变，就发出 PrefixCacheChange 事件并重新 pin baseline。

检测内容

fingerprint 包含 system prompt hash、排序后的 tool names、tool count 和 pinned time。

接入位置

turn_loop.rs 在构造 MessageRequest 前执行稳定性检查，然后再发送 streaming request。

输出价值

它把 prefix cache 变成可观测信号，而不是只等 provider usage 返回 hit/miss 后才发现问题。

局限

当前 fingerprint 只看 tool names，不 hash 完整工具 schema。若工具描述或参数 schema 改变但名字不变，检测可能不会报 drift。

这个局限很重要。CodeWhale 在工程上已经把 tool catalog 稳定性作为 cache 维度，但检测精度仍偏轻量，适合告警而不是严格证明前缀字节完全一致。

子 agent 复用

CodeWhale 的 subagent 有两个特别贴近 prefix cache 的设计。

机制 实现 缓存意义
fork_context 子 agent 复制父 runtime 的 system prompt 和 leading messages，再追加 <codewhale:fork_state>、<codewhale:subagent_context> 与子任务。 对同一父上下文分叉多个视角时，尽量保持父请求前缀字节相同。
resident_file spawn 时读取文件内容，作为固定前缀塞进 child prompt，并用全局 lease 防止两个 agent 同时持有同一文件。 适合“围绕同一大文件连续追问”的场景，让文件内容在 child session 里更稳定。
completion sentinel 子 agent 完成后只把短 summary 和轻量 <codewhale:subagent.done> 注入父循环。 避免把完整 child transcript 塞进父上下文尾部，降低 cache-miss tail 膨胀。

压缩与循环

CodeWhale 的默认策略偏保守: V4 replacement compaction 不因 reasoning effort 降低阈值；自动 replacement compaction 在配置层是 opt-in；Flash seam manager 和 capacity controller 也默认不介入。配置文档明确说 prompt cache hit/miss 只用于显示和成本估算，不能驱动 compaction、seam 或 cycle trigger。

上下文窗口

models.rs 把 V4 Pro/Flash 视为 1M context，默认压缩阈值约为窗口的 80%。

500K floor

compaction.rs 给自动压缩设置 500K 最小门槛，因为压缩会重写稳定前缀并破坏 KV cache。

cycle restart

cycle_manager.rs 针对 256K+ 之后的检索退化，提出 checkpoint-restart: 归档旧 cycle，带结构化工作状态开启新 cycle。

capacity controller

容量控制器默认关闭。文档写明主动干预可能改写 live prompt，破坏 prefix-cache affinity。

这套策略背后的判断是: V4 长上下文里的 token 成本不只看输入长度，还要看 cache 命中率和上下文质量。频繁压缩会让请求更短，但可能让稳定前缀反复失效。

Thinking 与工具输出

DeepSeek V4 thinking 模式下，工具调用的 assistant message 可能需要 reasoning_content。CodeWhale 在 client/chat.rs 里做 provider-aware sanitizer: 当历史 assistant 有 tool_calls 但缺 reasoning content 时，插入占位内容，并把 replay token 记录进 usage。

工具输出方面，truncate.rs 对超过阈值的大输出做 spillover: 完整内容写入 ~/.codewhale 优先，回退到 ~/.deepseek，上下文里只保留 head 和 artifact reference。这比简单截断更适合 agent 继续工作，因为模型知道完整输出在哪里，而不是误以为输出本身完整。

成本统计

CodeWhale 解析 provider usage 时显式区分 input/output、cached tokens、prompt cache hit/miss、reasoning tokens 和 reasoning replay tokens。pricing.rs 再按 DeepSeek V4 Pro/Flash 的 hit/miss 价格计算估算成本与 cache savings。

工程含义: cache telemetry 是反馈面板和成本仪表，不是自动删上下文的触发器。

这也是 CodeWhale 和很多 coding agent 的差异点: 它把 prefix cache 作为“编排稳定性”问题处理，而不是只在 UI 里显示一行 saved tokens。
