沙箱、审批与真实权限边界

CodeWhale 的安全模型分成三层: 模式决定工具暴露和 sandbox policy，审批决定单次高风险工具是否执行，平台 sandbox 决定命令进程能否被 OS 约束。

模式与工具暴露

tool_setup.rs 明确把 UI mode 映射成 sandbox policy。Plan 是 ReadOnly，Agent 是 workspace-write 加 network，YOLO 是 DangerFullAccess。

模式 工具目录 沙箱策略 工程含义
Plan 只读文件、搜索、git history、diagnostics、skill、validation、handle、runtime read-only task、todo/plan/goal。 ReadOnly 计划模式不暴露 shell、code execution、patch。它不是“执行但不批准”，而是目录层面拿掉危险工具。
Agent 完整 agent 工具、RLM/FIM、patch、web、shell 等按 feature 和 allow_shell 启用。 WorkspaceWrite，workspace writable，network allowed。 默认工作模式，依赖审批和平台 sandbox 约束写入、网络、命令。
YOLO 高权限工具可直接运行。 DangerFullAccess 明确无护栏契约，不应被描述为安全执行模式。

审批流

每个工具在 plan 阶段会被标注是否需要审批。非只读 MCP、代码执行、写入、agent_open 等会触发 ApprovalRequired，用户可以批准、拒绝，或 retry with elevated sandbox policy。

平台沙箱

sandbox/mod.rs 选择平台默认 sandbox: macOS 用 Seatbelt，Linux 检测 Landlock，Windows 当前不应宣传 OS 文件系统隔离。策略本身由 SandboxPolicy 表达，包括 read-only、workspace-write、danger-full-access 和 external-sandbox。

macOS

实际通过 /usr/bin/sandbox-exec 包装命令，设置 DEEPSEEK_SANDBOX=seatbelt，策略文件由 readable/writable roots、network 等规则生成。

Linux

如果配置 prefer_bwrap=true 且 /usr/bin/bwrap 存在，则用 bubblewrap 包装命令。否则当前常规路径只设置 DEEPSEEK_SANDBOX=landlock 标记。

Process hardening

Linux TUI 启动时调用 apply_process_hardening()，设置 PR_SET_DUMPABLE=0、PR_SET_NO_NEW_PRIVS=1 和 RLIMIT_CORE=0。

Windows

配置文档明确说 Windows 当前不宣传 OS sandbox。规划中的 helper 首先是 process-tree containment，不是完整 AppContainer 或注册表隔离。

Linux 实现核对

Linux 是这次调研最需要谨慎表述的部分。仓库里确实有 landlock.rs、seccomp.rs 和 process_hardening.rs。其中 LandlockSandbox 和 seccomp BPF filter 都有相当完整的代码。但是当前 SandboxManager::prepare_landlock 没有把命令交给 Landlock helper，也没有调用 LandlockSandbox::apply 或 apply_seccomp_filter。

关键结论

在当前快照里，Linux shell 命令的强隔离路径是可选 bwrap；没有 bwrap 时，常规路径更像“标记为 landlock 的未隔离执行”。process hardening 保护的是父 TUI 进程和后代权限提升边界，不等价于 per-command 文件系统 sandbox。

能力 代码状态 应如何描述
Landlock syscall 封装 landlock.rs 有 ruleset、path rule、restrict_self。 存在模块和基础实现，但常规 prepare path 未实际调用。
seccomp filter seccomp.rs 有 raw BPF 和 apply_seccomp_filter()。 存在模块，但未发现命令执行路径调用。
bwrap prefer_bwrap=true 且 /usr/bin/bwrap 可用时构造 bwrap 命令。 当前 Linux 命令级强隔离的实际可用路径。
process hardening main.rs 启动早期调用。 防 ptrace、core dump 和 privilege escalation，属于 defense-in-depth。

文档漂移

docs/SANDBOX.md 对 Linux 的描述比 README 和实际常规路径更强，写到了 Landlock、seccomp、process hardening 组合 enforcement。与此同时，README 和配置文档更保守，强调 Linux helper-gated、Windows 不宣传 OS sandbox。

因此调研结论采用“以源码路径为准”的口径: macOS Seatbelt 是实装；Linux process hardening 是实装；Linux bwrap 在满足配置和环境时实装；Landlock/seccomp 是模块存在但命令路径未真正接入。

复用建议

可以借鉴

模式层面收缩工具目录、工具层面审批、执行层面平台 sandbox、写入前 snapshot、执行后 LSP diagnostics，是一套清晰的 agent 安全分层。

不要误用

不要把 DEEPSEEK_SANDBOX=landlock 当成 Linux 文件系统强隔离证据。需要生产级 Linux 隔离时，应强制 bwrap、外部 sandbox backend，或补齐 Landlock/seccomp helper 并做端到端测试。
