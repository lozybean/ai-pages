入口与边界

仓库 README 把安装模型讲得很明确: npm wrapper 负责下载 release binary，真正运行的是 Rust binary pair。codewhale 是 dispatcher，codewhale-tui 是 companion runtime。

docs/ARCHITECTURE.md 也提醒当前架构拆分还在进行中，crates/tui 仍是 live runtime source of truth。因此分析编排时，应优先看 crates/tui/src/core/engine、crates/tui/src/client、crates/tui/src/tools，而不是只看已拆出的 crate 名称。

主循环阶段

Engine::handle_deepseek_turn 是最重要的调度点。文件头注释直接说明它负责 per-turn orchestration: request construction、streaming event handling、tool planning/execution、LSP post-edit hooks、capacity checkpoints 和 loop termination。

handle_deepseek_turn()
  |
  |-- clone client, init loop guard and tool catalog
  |-- drain mid-turn steer input
  |-- refresh system prompt and max steps
  |-- optional auto compaction and cycle checks
  |-- token budget preflight and overflow recovery
  |-- flush pending LSP diagnostics
  |-- build active tools and strict tool mode
  |-- resolve reasoning effort
  |-- check prefix-cache stability
  |-- create streaming MessageRequest
  |-- parse stream into content, thinking, tool_uses
  |-- if no tools: finish, wait, or continue
  |-- if tools: plan batches, approve, execute, compact results
  |-- run post-edit LSP hook, append ToolResult, continue

这个循环不把工具执行当作一次性副作用。每个工具调用先进入计划阶段，带有 interactive、approval_required、supports_parallel、read_only 等元数据，再决定是否并行、串行、阻断、自动 hydrate deferred tool，或触发审批。

Streaming 解析

CodeWhale 走 OpenAI-compatible Chat Completions streaming，而不是 Responses API。client/chat.rs 在请求里启用 stream_options.include_usage=true，并解析 DeepSeek 返回的 reasoning_content、工具调用 delta 和 usage-only chunks。

超时与重试

streaming.rs 设置 per-chunk idle timeout、wall-clock cap、最大 content bytes、最大 stream errors 和透明重试预算。只有在没有收到有效内容时才做透明重试。

thinking/reasoning

DeepSeek V4 thinking 模式对带 tool_calls 的 assistant message 要求 reasoning_content。CodeWhale 会在必要时插入占位 reasoning content，并记录 replay token。

工具片段

streaming state 维护 block kind、current tool indices、raw/visible text、tool input JSON buffer 和 fake wrapper scrubber，以容忍模型输出的多种 tool-call 形态。

工具规划与执行

工具系统的编排比“模型叫工具，宿主执行”多了几层保护。

阶段 机制 意义
注册 Plan 模式只暴露只读文件、搜索、git history、diagnostics、skill、validation、handle、runtime read-only task、todo/plan/goal。 Plan 不是靠审批阻止写入，而是在工具目录层面不暴露 shell 和 patch。
规划 解析 interactive、caller allow policy、MCP 只读/并行属性、tool_search hydration、loop guard。 先把“能不能执行”和“怎样执行”固定下来，减少模型输出歧义。
并行 并行批只运行 read-only 且 supports_parallel 的工具，结果按原始 index 写回 outcomes。 读操作加速，但模型可见的结果顺序保持稳定。
审批 写入、代码执行、非只读 MCP、subagent open 等触发 ApprovalRequired，支持 approve、deny、retry with elevated policy。 审批与沙箱 policy 联动，允许用户按单次工具调用提升权限。
副作用前 write_file、edit_file、apply_patch 前做 workspace snapshot。 为局部 undo 和审计留下恢复点。
副作用后 大输出 spillover、working set observe、post-edit LSP diagnostics。 避免长输出污染上下文，同时把编辑后的诊断反馈回下一步。

子 agent

CodeWhale 的子 agent 不是一次性 prompt 调用，而是 persistent background session。模型面对的主要表面是 agent_open、agent_eval、agent_close，旧的 agent_spawn 等路径仍保留兼容。

后台生命周期

background_runtime() 会为子 agent 创建独立 cancellation token。父 turn 取消不会自动杀掉已经打开的后台子 agent，显式关闭走 agent_close。

深度与并发

默认 spawn depth 为 3；并发上限默认 10，配置钳制到 1..=20。running agent 才计入上限。

上下文模式

fork_context=false 时是 fresh child；true 时尽量保留父请求的 system 和 leading messages 字节前缀，再追加结构化状态和子任务。

完成哨兵

直接子 agent 完成后向父循环发送 <codewhale:subagent.done> sentinel。sentinel 刻意保持很瘦，完整 transcript 通过 handle 或 agent_eval 拉取。

子 agent role 包括 general、explore、plan、review、implementer、verifier、tool_agent、custom。默认 agent 已从早期 allowlist 模型转向继承父 registry，但审批门和角色限制仍会约束高风险工具。

模型与推理路由

--model auto 和 reasoning_effort=auto 是两套相关但不同的机制。TUI 的 auto model router 会拿用户最新输入和最近上下文窗口，调用 flash router 选择 concrete model 与 reasoning effort；固定模型下的 auto thinking 则走本地启发式，不启动隐藏 router 请求。

子 agent 也有自己的路由逻辑。tool_agent 被强制路由到 deepseek-v4-flash 且 thinking off；其他子 agent 先看显式 model，再看 role/type 配置，最后继承父模型，必要时再咨询 flash router。
