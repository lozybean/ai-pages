调研口径

本次调研对象是 CodeWhale 公开仓库。调研前已同步远程 main 与 tags，分析基于源码快照 54151a4bc942af69f55298cae986858cd02100d1。

CodeWhale 自称是面向 DeepSeek V4 的 terminal coding agent。它不是 Node agent，npm 包更像安装器，实际运行时是一组 Rust 二进制: codewhale dispatcher 加 codewhale-tui companion runtime。

核心结论

编排核心

活跃运行时代码仍集中在 crates/tui。每轮由 handle_deepseek_turn 驱动，串起系统提示刷新、token 预算、prefix-cache 稳定性检查、OpenAI-compatible streaming、工具规划、审批、并行/串行执行、LSP 后置诊断和上下文控制。

Prefix Cache

CodeWhale 把 DeepSeek V4 的 1M context 和 prefix cache 当作主架构约束: 默认晚压缩、保持历史 turn metadata 不重写、forked subagent 复用父请求前缀，并通过 cache hit/miss 计算成本。

沙箱边界

macOS Seatbelt 路径是明确执行的。Linux 有 Landlock、seccomp、process hardening 和 bwrap 代码，但当前常规 Landlock 执行路径只设置标记，真正命令级隔离依赖可选 bwrap 或后续 helper。

远程通道

runtime API 是 local-first 的 HTTP/SSE API，默认绑定 127.0.0.1:7878，当前代码会在无 token 时生成临时 bearer token。远程方案主要是 Tencent Lighthouse 加 Feishu/Lark 长连接桥，而不是把 /v1/* 暴露到公网。

CodeWhale 的工程取向是“把编码 agent 做成本地 runtime”。TUI 是主入口，但同一套 runtime 还向 HTTP/SSE、ACP、任务队列、automation、Feishu/Lark bridge 和 app-server 暴露能力。

关键判断

值得借鉴

CodeWhale 对 DeepSeek V4 的适配比普通 OpenAI-compatible wrapper 更深: 它处理 thinking/reasoning content、prefix-cache 稳定性、1M context、cache hit/miss 成本、auto model routing 和子 agent 前缀复用。

需要谨慎

仓库正在拆分 crate，文档和实现存在局部漂移。尤其是 Linux 沙箱，文档描述比当前常规执行路径更强。调研或复用时应以 crates/tui 活跃路径为准。
