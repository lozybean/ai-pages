调研口径

本次调研对象是用户指定的 Reasonix 仓库，而不是其他 DeepSeek agent 项目。分析基于公开仓库源码快照 1095f5ff52946c07449b4faa7b0cc5dca17477f7。

Reasonix 的自我定位是 “DeepSeek-native coding agent”。它不是把通用 OpenAI SDK 包一层 UI，而是围绕 DeepSeek 的 prefix cache、tool-call 可靠性和低成本长上下文循环做了专门设计。

核心结论

编排核心

主循环由 CacheFirstLoop 驱动，状态拆为 immutable prefix、append-only log、volatile scratch。一次 step 中串起预算检查、上下文折叠、模型流式响应、tool-call 修复、工具调度和持久化。

Prefix Cache

优化重点不是“少发上下文”，而是让昂贵、稳定的前缀字节在每轮请求中保持完全一致。工具规格、系统提示和 few-shot 固定在 prefix，动态内容只追加或折叠。

沙箱边界

Reasonix 没有实现 kernel/OS 级沙箱。它的安全模型是应用层 rootDir 限制、路径审批、shell allowlist、敏感路径降级、edit gate、plan mode 和 PauseGate。

远程通道

仓库没有内置 cloudflared/ngrok 类反向隧道。远程能力主要来自本地 dashboard HTTP server、QQ Bot WebSocket channel，以及面向 host 集成的 ACP stdio/JSON-RPC。

Reasonix 的设计重心很清楚：把稳定、昂贵、会影响 cache key 的内容固定住，把不可避免的动态内容推到 append-only log 或 scratch 中，把工具调用失败的概率在本地修复掉，并用上下文折叠维持长会话。

工程判断

值得借鉴

Reasonix 把 DeepSeek prefix cache 当作架构约束，而不是价格优化小技巧。它对前缀稳定性、工具 schema 扁平化、流式 tool-call 拼接、结果顺序确定性和上下文折叠都有直接代码支撑。

需要保持清醒

“sandbox”主要是应用层约束。只要用户批准了命令，进程仍以当前用户身份运行；dashboard 若暴露到非 loopback，也只是 token 保护，没有内置 TLS 或反向隧道隔离。
