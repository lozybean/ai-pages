Dashboard、QQ 与远程通道

Reasonix 没有内置通用反向 tunnel。它提供的是本地 dashboard 控制面、QQ Bot 远程消息入口，以及 ACP stdio/JSON-RPC 集成协议。

结论

Reasonix 的远程能力分成三层：dashboard 是本地 HTTP 控制面；QQ channel 是把用户消息和 agent 回复映射到 QQ Bot WebSocket/HTTP API；ACP 是面向其他 host 程序的 stdio JSON-RPC 协议。

源码中没有看到 cloudflared、ngrok、frp、Tailscale Funnel 或自研 reverse tunnel。若需要跨网络访问 dashboard，推荐把 tunnel 当作外部部署层，而不是 Reasonix 内置能力。

Dashboard：本地 token-gated HTTP server

Dashboard server 默认绑定 127.0.0.1，端口可配置或自动选择。启动后 URL 带 ?token=...，token 可显式提供，也可由配置生成 32-byte token。

  • 读请求接受 header token 或 query token。
  • mutation 请求要求 X-Reasonix-Token header，用于降低 CSRF 风险。
  • SPA HTML、assets、SSE events、API 都在 token gate 之后。
  • 非 loopback host 会给出警告，提示这会把控制面暴露给其他设备。
Dashboard = local control plane + bearer-like token + SSE event stream + modal resolution APIs

远程审批：dashboard 与 TUI 共用同一 PauseGate

Dashboard 不是只读面板。它可以拿到 active modal snapshot，并通过 /api/modal/resolve 解析 shell、path、choice、plan、edit-review、checkpoint、revision、picker、viewer 等交互。

这与 Reasonix 的权限模型是一体的：本地 TUI 看到的确认弹窗，dashboard 也能镜像并解析。优点是移动端或浏览器远程操作完整；风险是 dashboard token 泄漏时，攻击者也可能解析这些确认。

mutation 要求 header token，但这不是强认证系统。不要把 dashboard URL 公开发布，也不要在没有 TLS 的公网链路中传递 token。

QQ Channel：远程消息入口，而非独立 agent

QQ 文档和代码都强调：QQ channel 是 attach 到已有 chat/code/desktop session 的远程通道。它向本地 session 提交 [QQ] ... 用户消息，并把 Reasonix 回复分块发送回 QQ。

连接方式

通过 QQ Bot access token 和 gateway WebSocket 建连。gateway URL 必须是 wss，且 host 在可信 QQ 域名集合中。

单例锁

使用 ~/.reasonix/qq-channel.pid 做单例锁，避免同一个本地 runtime 被多个 channel 重复驱动。

访问控制

支持 owner、allowlist 和 runtime-bound first sender。若未配置 owner/allowlist，首个发送者会绑定 runtime。

消息限制

QQ 回复会按 1500 byte 限制分块，并在 markdown 发送失败时 fallback 到普通文本。

QQ channel 可以继续确认、选择、checkpoint、plan 等互动，这让它更像“远程控制通道”，而不是单纯通知通道。

ACP：面向 host 集成的 stdio JSON-RPC

ACP server 是 NDJSON JSON-RPC 2.0：一行一个 JSON 对象，通过 stdin/stdout 传输。它支持 request、notification、outbound request 和响应关联。

这条路径更适合 IDE、桌面宿主或其他进程把 Reasonix 嵌进去。它不是网络 tunnel，而是本机进程间协议；权限请求可以通过 ACP gate 映射到 host 侧的 permission UI。

Tunnel 口径：Reasonix 没有内置反向隧道

如果产品文案里提到从手机、浏览器或 QQ 继续会话，需要区分三件事：

能力 Reasonix 内置 备注
本地 dashboard 是 HTTP server，默认 loopback，token gate。
LAN/SSH tunnel 访问 部分 可配置 host/port，但 tunnel 由外部网络层提供。
QQ Bot 远程入口 是 通过 QQ 官方 WebSocket gateway 和 HTTP API。
通用公网 reverse tunnel 否 未发现 cloudflared/ngrok/frp 类实现。

部署建议

  • Dashboard 保持默认 127.0.0.1，需要手机访问时优先用 SSH tunnel 或可信 LAN。
  • 如果必须绑定 0.0.0.0，使用长随机 token，并放在受控网络内。
  • QQ channel 建议配置 owner 或 allowlist，避免 first-sender 绑定在初始化阶段被错误用户抢占。
  • 把 dashboard token 当作高敏凭据处理，不写进公开文档、截图或聊天记录。
