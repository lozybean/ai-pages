Prefix Cache 优化机制

Reasonix 的成本优化不是只靠便宜模型，而是把 DeepSeek 的自动 prefix cache 当作协议约束来维护：稳定前缀固定，动态内容追加，临时 scratch 不污染下一轮。

为什么有效

DeepSeek 的 prefix cache 依赖请求前缀字节保持一致。Reasonix 的关键判断是：工具规格、系统提示、few-shot 和长会话历史一旦被重排、重序列化或塞入不稳定 scratch，就会把 cache key 打碎。

因此 Reasonix 把“稳定字节”变成工程不变量：prefix 在会话开始时固定；后续对话只 append；reasoning 与临时状态放到 volatile scratch；到达上下文阈值时通过 summary folding 替换旧日志，而不是改写 prefix 本身。

三段式状态

ImmutablePrefix

包含 system prompt、tool specs 和 few-shot。源码注释明确说工具 add/remove/replace 会 invalid fingerprint，并承认工具变化会导致一次 cache miss。

AppendOnlyLog

对话历史只暴露 append/extend。压缩和恢复用 compactInPlace 走显式路径，避免常规 turn 中对历史做原地重排。

VolatileScratch

存放 reasoning 和 turn-local 状态。它能让模型思考内容不进入下一轮稳定前缀，降低 cache 污染和 token 膨胀。

Fingerprint

prefix fingerprint 会 hash system、完整 tool specs 和 few-shot。这比只 hash 工具名更严格，也更符合 prefix cache 的字节级敏感性。

上下文折叠：用 summary 换上下文空间

ContextManager 在 prompt usage 接近上限时触发折叠。默认阈值包括 fold、aggressive fold、force summary 和 turn-start fold。折叠时保留尾部最近消息，把旧消息总结为 synthetic summary，再替换 log。

重要的是，fold summary 的模型调用复用当前 live prefix：system、tools、few-shot 仍是同一组稳定字节。这样 summary 过程本身也尽量共享 cache，而不是另起一套 cache-hostile 请求。

prefix messages + healed head messages + summarization instruction -> synthetic summary + recent tail

折叠提示还会保留 pinned skills、用户约束和未完成状态，避免压缩后丢掉 agent 工作契约。

输出瘦身：减少会话日志中的大块漂移

Reasonix 在多个位置主动瘦身，目标是减少下一轮 prompt 中的低价值大块文本：

  • shrinkOversizedToolRole 和 shrinkToolRoleToTokenCap 会压缩工具结果，尤其是长输出。
  • shrinkAssistantToolArgs 会压缩 assistant 中过大的 tool-call 参数，保留 key、路径和短值。
  • read_file 对大文件默认返回 outline、head 和符号地图，而不是一次性塞入完整文件。
  • 依赖目录默认跳过，避免 node_modules、构建产物等把上下文拖入低价值区域。

这里的思路不是简单截断，而是把“模型下一步需要什么”作为返回结构设计目标：小文件给全文，大文件给结构和下一步可执行的定位建议。

成本统计：cache 命中是第一等指标

DeepSeek client 会把 API 返回的 cache hit/miss token 写入 Usage，统计层再计算单 turn 和累计 cache hit ratio、成本、输入成本、输出成本、节省估算。

模型 Cache hit 输入价 Cache miss 输入价 输出价
deepseek-v4-flash $0.0028 / 1M $0.14 / 1M $0.28 / 1M
deepseek-v4-pro $0.003625 / 1M $0.435 / 1M $0.87 / 1M

这组价格让 hit/miss 不是小优化：对 flash 来说，cache miss 输入价约是 cache hit 输入价的 50 倍。

实测案例：99.82% input cache hit

仓库内的 real-world cache benchmark 记录了一个用户在 2026-05-01 的 DeepSeek dashboard 截图数据：cache hit input 为 435,033,856 tokens，cache miss input 为 767,616 tokens，input cache hit ratio 为 99.82%。

项目 Reasonix 案例 同工作量 0% cache
flash 日成本 $1.38 $61.06
节省 约 $59.68，约 97.7%

这个 benchmark 是项目方提供的匿名真实案例，不应泛化成所有用户都能达到 99.82%。但它证明了 Reasonix 的架构目标是可被观测到的：命中率来自客户端保持 prefix 稳定，而不只是 DeepSeek 服务端“自动支持 cache”。

可复用建议

  • 把 system prompt、工具 spec、few-shot 当作协议前缀，不要在每轮动态拼装或改变排序。
  • 工具结果必须结构化收缩，不能把完整 stdout、完整大文件、完整 dependency tree 默认塞回消息日志。
  • 上下文折叠要有明确阈值，并把“保留约束、保留未完成状态、保留尾部上下文”写进 summary contract。
  • 在 UI 中直接展示 cache hit/miss token 与成本，让 prefix drift 可以被用户和开发者及时发现。
