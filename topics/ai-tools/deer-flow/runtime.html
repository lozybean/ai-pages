DeerFlow 研究：Run Runtime 与运行一致性

从一次 Run 的创建、执行、流式输出、取消、回滚和状态写入出发，梳理 DeerFlow 如何把用户交互转化为可观察、可中断、可持久化的运行过程，并说明当前实现的单 worker 边界与分布式一致性要求。

核心模型

Run Runtime 处理的不是“一次请求返回一个回答”，而是“一次可被观察、可被取消、会写入状态的执行”。在 DeerFlow 中，Thread 承载持续上下文，Run 承载一次执行尝试，RunRecord 记录执行生命周期，Worker 推进 agent 图，Checkpointer 保存 ThreadState，StreamBridge 把执行过程投影为 SSE 事件。

理解 Run Runtime 的关键，是把事实来源和观察通道分开：checkpoint 是继续对话的事实来源，RunRecord 是执行生命周期记录，StreamEvent/SSE 是客户端观察一次执行过程的通道。

整体模型：定位 Run 的位置

在展开取消、回滚、用户主动打断和排队之前，需要先把 Run 放回 DeerFlow 的整体模型里。Run 既不是 Thread 本身，也不是 SSE 连接；它位于持续上下文和一次执行之间：从 ThreadState 读取上下文，驱动 Agent 执行，再把过程投影成 StreamEvent，并把结果写回 checkpoint。

概念 在 Run Runtime 中的定位 实现承载 边界判断
Thread 持续会话容器，是多次 Run 共享的上下文和权限边界。 thread_id、LangGraph checkpoint、ThreadMetaStore。 一个 Thread 可以有多个 Run；Thread 不是一次执行，也不是一次 SSE 连接。
ThreadState Run 开始和结束时围绕它读写的图状态。 ThreadState 细化 AgentState，包含 messages、title、todos、artifacts、thread_data、sandbox 等。 values 事件只是 ThreadState 的快照投影；真正的恢复来源是 checkpointer。
Run 对某个 Thread 发起的一次执行尝试，是本文分析的核心对象。 Gateway Run Service、RunManager、RunRecord、worker task。 Run 不等于完整对话。失败、重跑、用户补充打断都会形成新的 Run。
RunRecord Run 的生命周期事实记录。 run_id、thread_id、status、metadata、task、abort_event、model name、token usage。 它回答“这个 Run 是否活着、如何取消、最终状态是什么”；不保存完整 ThreadState。
Agent Execution 一次 Run 内部真正推进模型、工具和 middleware 的执行过程。 worker 调用 agent.astream()，Agent factory 组装 model、tools、middleware、checkpointer、store。 Agent 负责推进 state；Run Runtime 负责生命周期、取消、回滚和事件投影。
StreamEvent Run 执行过程的观察事件。 metadata、values、messages、error、end 等事件。 事件用于前端观察和重连，不是 Thread 的唯一事实来源。
StreamBridge / SSE 把 worker 生产的事件交给客户端消费。 publish、subscribe、heartbeat、Last-Event-ID replay、StreamingResponse。 SSE 是传输连接；断连是否取消 Run 取决于 on_disconnect。

Run 的语义

Run 是 Thread 上的一次执行尝试。普通用户输入、resume、wait、stream、主动打断后的补充消息，本质上都会变成新的 Run。

容易误解的地方

Run 不是完整对话，也不是可以随意迁移的后台任务。当前实现里，一个正在运行的 Run 绑定本进程内的 asyncio.Task、abort_event 和 memory stream buffer。

Runtime 协作模型

Run Runtime 的组件可以分成三层：入口层负责协议和权限，执行层负责生命周期和 worker task，持久层负责状态、事件和元数据。这样看，很多交互功能都不是单点能力，而是这些组件共同完成的一次状态迁移。

组件 核心职责 一致性含义
RunManager 创建 Run、维护本进程 active run 索引、执行 reject / interrupt / rollback 策略。 当前 inflight 判断是本进程视角。多节点部署需要额外的 thread 级全局互斥。
RunRecord 记录 status、task、abort_event、abort_action、token 和错误。 task 和 abort_event 是内存对象，不能跨 worker 直接使用。
Worker 运行 run_agent()，调用 agent.astream()，把 chunk 转成 StreamEvent。 真正的取消发生在 worker 对 CancelledError 和 abort_event 的响应里。
Checkpointer 保存和读取 ThreadState，是下一次 Run 继续执行的事实来源。 跨节点续接依赖共享 checkpointer；默认内存 checkpointer 不具备这个能力。
StreamBridge 解耦 worker 事件生产和 SSE 消费，支持 bounded replay 和 heartbeat。 当前 memory bridge 只在本进程有效，实时 join stream 有 worker 亲和性。

RunRecord 的重要字段

RunRecord 是 Run Runtime 的最小执行账本。它不保存完整对话状态，而是记录一次 Run 如何被识别、归属、取消、观察和结算。理解这些字段后，后面的用户主动打断、取消、rollback 和断连行为会更清晰。

字段 作用 运行时含义
run_id 一次执行尝试的唯一标识。 SSE Content-Location、join stream、cancel、RunStore 查询都围绕它定位一次 Run。
thread_id Run 所属 Thread。 决定读取和写入哪个 checkpoint，也决定同一 Thread 下并发策略如何判断 active run。
assistant_id / model_name 记录本次执行使用的 agent 和模型线索。 创建 agent、展示 run 元数据、统计 token 和追踪时使用。实际模型名可能在 agent 构建后被修正。
status Run 生命周期状态。 pending、running 是活跃态；success、error、interrupted 是终态。
on_disconnect SSE 客户端断开时的处理策略。 cancel 会在 SSE consumer 收尾时取消后台 Run；continue 允许 Run 继续执行并写 checkpoint。
multitask_strategy 同一 Thread 已有 active Run 时，新 Run 的处理策略。 当前 RunManager 支持 reject、interrupt、rollback；enqueue 只在 API schema 中出现，运行时未实现。
task 本进程内执行 run_agent() 的 asyncio.Task。 这是 cancel 和 shutdown 能直接作用到 worker 的关键字段；跨进程或跨节点不可用。
abort_event / abort_action 协作式取消信号和取消动作。 worker 在流式循环中检查 abort_event；abort_action 决定是普通 interrupt 还是 rollback。
store_only 表示该记录是从 RunStore 恢复出来的只读记录。 它没有本地 task 和 abort_event。当前代码会拒绝对 store-only Run 做 join stream；取消也无法直接作用到真实 worker。
token / message 字段 记录输入输出 token、模型维度 token、LLM 调用次数、消息数量等。 由 RunJournal 和完成阶段更新，服务运行统计、列表展示和成本分析。

Run 的生命周期状态轮转

Run 的状态机可以分成两段：创建与执行阶段，以及终态结算阶段。pending 和 running 是活跃态；success、error、interrupted 和 timeout 是终态。当前 worker 主流程会主动落到 success、error 或 interrupted，timeout 是 schema 和数据库模型中的状态值，但主执行链路没有直接把 Run 设置为 timeout。

状态 进入条件 后续行为
pending RunManager.create() 或 create_or_reject() 创建 RunRecord，并写入 RunStore。 等待 worker task 启动；如果此时被 interrupt 或 rollback 抢占，也会进入取消路径。
running worker 进入 run_agent() 后调用 set_status(running)。 持续产出 stream event、写 checkpoint、刷新 progress，直到完成、异常或收到取消信号。
success agent.astream() 正常结束，且没有 LLM fallback error，也没有 abort。 worker flush journal，写 completion，更新 ThreadMeta 状态为 idle，发布 end。
interrupted 显式 cancel、SSE 断连触发 cancel、或新 Run 以 interrupt 策略抢占旧 Run。 保留已经写入的 checkpoint；旧 Run 进入终态，新用户意图应通过新的 Run 表达。
error worker 捕获异常、LLM fallback error，或 rollback 策略触发回滚并把旧 Run 标记为错误。 错误信息写入 RunStore；rollback 会尝试恢复 pre-run checkpoint。
timeout 当前 schema 和数据库模型保留了该状态。 当前主执行链路没有直接设置该状态；客户端等待超时不等同于 Run 进入 timeout 终态。

RunManager 的主要公开方法

RunManager 同时承担本进程 active run registry 和可选 RunStore 持久化门面。它的方法可以按“创建与查询、状态迁移、取消与并发、恢复与清理”理解。

方法 作用 关键边界
create() 创建一个 pending Run，并写入内存索引和 RunStore。 如果持久化新 Run 失败，会回滚内存记录，避免调用方看见一个不可持久化的 Run。
create_or_reject() 创建 Run，同时处理同一 Thread 上的 active run。 reject 发现 inflight 就报冲突；interrupt/rollback 会设置旧 Run 的 abort_event 并 cancel task。
get() / aget() 按 run_id 查询 RunRecord。 先查本地内存；查不到再从 RunStore hydrate。RunStore 恢复出来的是 store_only=True。
list_by_thread() 列出某个 Thread 下的 Run。 合并本地内存记录和 RunStore 记录，按创建时间倒序返回。
set_status() 更新本地 RunRecord 状态，并尽力同步到 RunStore。 只作用于本地已知 Run；如果 record 不在本进程内存中，会记录 warning 并返回。
update_model_name() 修正本次 Run 实际使用的模型名。 agent 构建后可能解析出默认模型或替代模型，需要把最终结果落到 RunRecord/RunStore。
update_run_progress() 运行中刷新 token、消息等进度快照。 只在 record 仍是 running 时更新，避免终态后进度写入覆盖完成结果。
update_run_completion() Run 结束时持久化 token、消息、最后 AI 消息、首条用户消息等结算数据。 由 worker 的 finally 阶段调用，属于完成收尾而不是流式事件本身。
cancel() 请求取消一个本地 active Run。 设置 abort_action、abort_event，并对本地 task 调用 cancel()。找不到本地 record 或状态不可取消时返回 false。
reconcile_orphaned_inflight_runs() 启动时把没有本地 worker 持有的 persisted inflight Run 标记为错误。 当前用于避免重启后 UI 永远显示 running。它不是实时 worker health detection，也不是“迁移并恢复旧 worker”；通常要等启动恢复或扫描流程发现 persisted inflight 后，才会把孤儿 Run 变成明确错误。
has_inflight() 判断本进程内某个 Thread 是否有 pending/running Run。 这是本地视角，不是多节点全局互斥。
cleanup() / shutdown() 清理本地 RunRecord，或在进程退出前 drain/cancel 活跃 task。 shutdown 会在 checkpointer 关闭前尽量等待 Run 收尾，降低 checkpoint 写入撞上关闭连接池的风险。

Worker 与 RunManager 的关系

Worker 不是一个独立可调度实体，而是 start_run() 里通过 asyncio.create_task(run_agent(...)) 创建出来的本地后台任务。RunManager 创建并持有 RunRecord，Worker 在执行过程中反过来调用 RunManager 更新状态、模型名、进度、完成统计和取消结果。

当前支持的关系

同一个进程内，RunManager 可以直接通过 record.task.cancel() 打断 Worker；Worker 可以通过 record.abort_event 感知取消，并根据 abort_action 执行 interrupt 或 rollback。

当前不支持的关系

DeerFlow 当前不支持把一个正在运行的 Worker 迁移到另一个节点后继续执行。RunStore 可以恢复 RunRecord 的持久化字段，但恢复出来的是 store_only 记录，没有本地 task、abort_event 和 stream buffer。

这意味着，如果 Worker 所在进程退出或请求打到另一个节点，系统不能“重新开启同一个 Worker”。当前更接近的处理是：启动时把孤儿 inflight Run 标记为 error，用户或上层系统再基于同一个 Thread 创建新的 Run，从 checkpoint 继续或重跑。

如果要优化成分布式可恢复模型，需要把 worker ownership 从内存对象提升为持久协议：RunRecord 增加 worker_id、lease、heartbeat、fencing token；取消意图持久化为 cancel_requested 和 cancel_action；调度器基于 Thread 级锁启动新 worker；实时事件从 MemoryStreamBridge 迁移到 Redis Streams 或消息队列。即便这样，恢复的通常也应是“创建一个新的执行尝试继续 Thread”，而不是复活已经消失的 Python task。

Checkpointer 与 StreamBridge 在主流程中的作用

Checkpointer 和 StreamBridge 都不是主控组件，但它们分别守住了 Run Runtime 的两个关键边界：状态恢复边界和过程观察边界。

组件 在主流程中的作用 在取消 / 回滚中的作用 分布式边界
Checkpointer Run 启动后被挂到 agent 上，LangGraph 执行过程中通过它保存 ThreadState；/wait 完成后也会读取它返回最终 state。 Worker 在 Run 开始时捕获 pre-run checkpoint snapshot；rollback 会把该 snapshot 写回，或在无 snapshot 时删除 thread state。 跨节点延续依赖共享 checkpointer。memory checkpointer 只适合本进程；SQLite 更偏单机；Postgres 才更适合多节点共享状态。
StreamBridge Worker 发布 metadata、messages、values、error 和结束信号；SSE consumer 订阅并格式化为浏览器可消费的 SSE frame。 取消或异常后，worker 仍会在 finally 中 publish_end，避免客户端流无限等待。断连时是否取消 Run 由 SSE consumer 和 on_disconnect 配合决定。 当前 MemoryStreamBridge 是 per-process event log，只能支持本 worker 上的 join/replay。配置里有 redis 类型，但实现仍是未完成状态。

RunStore、RunEventStore、ThreadMetaStore 的建模边界

这三个 Store 都属于持久层，但它们保存的不是同一种事实。RunStore 记录一次 Run 的生命周期账本，RunEventStore 记录可查询的事件流和消息投影，ThreadMetaStore 记录 Thread 的列表元数据。完整 ThreadState 仍由 Checkpointer 负责，实时 SSE 分发仍由 StreamBridge 负责。

Store 建模边界 关键字段 关键能力
RunStore 保存 RunRecord 可序列化部分，用于 run history、状态恢复、列表展示、token 聚合和 orphan inflight reconciliation。 run_id、thread_id、assistant_id、user_id、status、model_name、multitask_strategy、metadata_json、kwargs_json、error、message_count、first_human_message、last_ai_message、token 统计、follow_up_to_run_id、created_at、updated_at。 put、get、list_by_thread、update_status、update_run_progress、update_run_completion、list_inflight、aggregate_tokens_by_thread。
RunEventStore 保存 Run 过程中的事件投影。它主要由 RunJournal 通过 LangChain callbacks 写入，用于历史消息、调试审计、run event 查询和消息分页，不承担实时 fan-out，也不替代 checkpoint。 thread_id、run_id、user_id、event_type、category、content、event_metadata、seq、created_at。其中 category 常见取值包括 message、trace、outputs、error、middleware 等，seq 在同一 Thread 内单调递增。 put、put_batch、list_messages、list_events、list_messages_by_run、count_messages、delete_by_thread、delete_by_run。
ThreadMetaStore 保存 Thread 的产品侧元数据，用于线程列表、标题、状态、归属和 metadata 搜索。它不保存 messages，也不保存 ThreadState。 thread_id、assistant_id、user_id、display_name、status、metadata_json、created_at、updated_at。 create、get、search、update_display_name、update_status、update_metadata、update_owner、check_access、delete。

RunStore 的边界

RunStore 能恢复 store_only RunRecord，用于查看历史和修正孤儿 running 状态；但它恢复不了本地 asyncio.Task、abort_event 和 StreamBridge buffer，所以不能靠它迁移一个正在执行的 worker。

RunEventStore 的边界

RunEventStore 的 seq 解决的是同一 Thread 内事件查询顺序。前端实时打字机仍主要消费 StreamBridge/SSE 的在线事件；历史消息、调试审计和部分重连补齐才更依赖事件存储。

ThreadMetaStore 的边界

ThreadMetaStore 更接近会话目录表。worker 结束时会从 checkpoint 里同步 title 到 display_name，并按 Run 结果更新 Thread 状态；这些写入是展示元数据，不是 Agent 状态本身，也不是严格的执行锁。

不要混用的事实来源

继续执行要读 Checkpointer；展示 run 列表要读 RunStore；展示历史消息要读 RunEventStore；展示 thread 列表和权限边界要读 ThreadMetaStore。把这些职责合并会让取消、rollback 和重连语义变得不可验证。

事件与状态的事实层关系

同一个 Run 的过程会同时出现在三类通道里，但它们不是同一个事实层。StreamBridge/SSE 面向在线观察，负责把 worker 产出的 metadata、messages、values 等事件推给前端；RunEventStore 面向持久查询，主要由 RunJournal 把 LangChain callback 标准化成 run events；Checkpointer 面向下一次执行，保存可恢复的 ThreadState。三者可以相互印证，但不能假设完全等价。

事实层 主要写入者 消费方式 不承担的职责
SSE / StreamBridge worker 在 agent.astream() 循环中 publish 在线事件。 前端实时渲染、打字机效果、运行过程观察。 不作为下一次 Run 的恢复事实来源。
RunEventStore RunJournal 基于 LangChain callbacks 写入 message、trace、lifecycle 事件。 历史消息分页、调试审计、run event 查询。 不负责实时 fan-out，也不保证和 SSE 每一帧一一对应。
Checkpointer LangGraph 执行过程中写入 channel values 和 checkpoint。 下一次 Run 加载 ThreadState、/wait 读取最终 state、rollback 恢复 pre-run checkpoint。 不表达完整 UI 流式过程，也不撤销外部副作用。

设计哲学

从 Runtime 协作模型可以看出，DeerFlow 没有把 Run 设计成一个可跨 worker 迁移和恢复的 durable job，而是把它定义为一次可观察、可中断、可审计的执行尝试。长期可恢复的是 ThreadState，短期执行现场则留在当前 worker 进程内。

DeerFlow 的核心取舍是：Run 是执行尝试，ThreadState 是长期事实；worker 失败后不复活原 Run，而是让失败可见，并通过新的 Run 基于 checkpoint 继续。

失败显式化

单机 worker 不保证分布式安全。一个 task 如果因为 worker、进程或本地运行时原因失去执行者，会被视为孤儿 inflight Run，启动恢复阶段会把它标记为错误，而不是让另一个 worker 接管同一个 Python task。

中断终态化

Run 的中断也是终态。用户从中断点继续，不是重启原 Run，而是创建新的 Run，并通过 checkpoint 加载 ThreadState。这样每次用户意图和每次执行尝试都有独立记录。

设计取舍 收益 代价
Run 不跨 worker 迁移 不需要 worker lease、heartbeat、fencing token、分布式 cancel channel 和跨节点 stream buffer，运行时实现更轻。 长任务遇到 worker 故障时不能原地恢复，只能把旧 Run 标记失败，再由用户或上层系统发起新 Run。
恢复 ThreadState，而不是恢复 task 现场 恢复边界落在 checkpoint 上，语义清楚：下一次执行从持久化状态继续。 模型调用栈、工具执行现场、未投递的实时 stream buffer 都不会被恢复。
中断作为终态 用户主动打断、系统取消、rollback 都能在 Run 历史中留下明确记录，便于审计和 UI 解释。 如果产品想做“暂停后原地继续同一个 Run”，当前模型不直接支持，需要额外的 durable execution 协议。
失败交给用户重新确认 避免系统自动重试造成重复工具调用、重复外部请求、重复写文件等副作用。 用户体验上会比自动恢复更保守，尤其是研究、生成、长链路任务中更容易感知失败。

与其他 Agent 运行框架的对比

框架类型 典型运行方式 与 DeerFlow 的差异
轻量 Agent Runtime 多数 Python Agent 框架直接在本地 coroutine、task 或请求线程里执行 agent。 它们通常也不保证 worker 级分布式恢复。DeerFlow 更进一步，把 RunRecord、RunEventStore、ThreadMetaStore 和 checkpoint 组合起来，让失败、中断和事件历史成为产品可见状态。
LangGraph Checkpoint 型 通过 thread/checkpoint 保存图状态，下一次执行可从状态继续。 DeerFlow 的恢复逻辑更接近这一类：恢复的是 ThreadState，不是恢复原 worker task。新的用户意图会创建新的 Run，并基于 checkpoint 继续推进图。
Durable Workflow 型 Temporal 这类系统把执行建模为 durable workflow，配套 activity retry、worker crash recovery、lease、heartbeat 和确定性 replay。 可靠性更强，但架构和开发约束更重。对 LLM 流式输出、非确定性模型调用、外部工具副作用，需要额外设计 replay、幂等和补偿机制。

因此，DeerFlow 当前更适合交互式 Agent 产品：用户可以观察执行、主动打断、补充信息、基于新 Run 继续。若目标变成长时间后台任务、强 SLA、多节点任意接管或自动恢复，Run Runtime 就需要向分布式调度器或 durable workflow 方向扩展。

主要交互过程

把 Run Runtime 放进真实交互里看，取消、回滚、断连和排队都不是孤立机制，而是不同的 Run 调度策略。它们共同遵循一个原则：新的用户意图通常创建新的 Run，旧 Run 是否保留、停止或回滚，由策略决定。

  • 普通流式执行

    普通流式执行的主线是：Gateway 创建 RunRecord，RunManager 注册 pending run，Run Service 启动 worker task，worker 调用 agent 图，StreamBridge 持续发布事件，SSE consumer 把事件推给客户端。

    • values 事件是 state 快照投影，不等于前端唯一事实来源。
    • Run 完成后，worker 在 finally 中 flush journal、同步 title、更新 thread 状态并发布 end。

  • 用户主动打断并补充信息

    用户主动打断不是 Agent 澄清。它的语义是：用户在观察 Run A 的过程中发现方向不对，向同一个 Thread 发送补充消息 M2。M2 应创建 Run B，并通过 interrupt 或 rollback 决定如何处理 Run A。

    • interrupt 适合“当前方向可修正，保留已产生上下文”。
    • rollback 适合“本轮方向错误，需要撤回到 Run A 开始前”。
    • 更严谨的实现应让 Run B 等 Run A 到达终止边界后再读取 checkpoint，避免 rollback 和新执行交错。

  • 显式停止与断连

    显式停止通常走 /runs/{run_id}/cancel 或 SDK 对 existing stream 的 POST stop 入口。SSE 断连则由 on_disconnect 决定：cancel 会请求取消后台 Run，continue 则允许 Run 在后台继续写 checkpoint。

    stop / cancel

    Gateway 调用 run_mgr.cancel()，设置 abort_action 和 abort_event，再对本地 asyncio.Task 调用 cancel()。

    disconnect

    SSE consumer 在 finally 中检查 Run 是否仍 active。如果策略是 cancel，则触发同样的取消路径；如果是 continue，worker 继续运行。

  • 重跑、排队与抢占

    失败 Run 不会被“复活”。重新执行应该创建新的 Run，并明确它基于哪个 checkpoint 继续。排队同样应建模为多个 Run 的串行调度，而不是把多个用户消息塞进同一个 running Run。

    策略 当前语义 适合场景
    reject 同一 Thread 有 active Run 时拒绝新 Run。 保守模式，避免并发写 ThreadState。
    interrupt 请求中断旧 Run，创建新 Run，保留旧 Run 已写状态。 用户补充信息、纠偏、改变要求。
    rollback 请求中断旧 Run，并恢复旧 Run 开始前 checkpoint。 用户认为本轮执行方向完全错误。
    enqueue API schema 中有字段，但当前 RunManager 未实现。 应扩展为 queued Run 和 thread 级调度器。

运行一致性边界

在“Run 是执行尝试，ThreadState 是长期事实”的设计哲学下，一致性边界不能全部理解为缺陷。更准确的分法是：一部分是 DeerFlow 有意选择的设计边界，另一部分才是实现和部署时需要警惕的风险。

当前模型的目标不是让一次 Run 永远可恢复，而是让每次执行尝试有明确终态，并把下一次执行建立在 checkpoint 上。

符合设计的边界

worker 现场不迁移

asyncio.Task、abort_event 和 MemoryStreamBridge 是进程内对象。worker 失败后标记旧 Run 失败，再由用户或上层系统发起新 Run，符合当前设计。

中断是 Run 终态

interrupted 不是暂停态。继续执行要创建新的 Run，并从 checkpoint 加载 ThreadState，这让用户意图和执行尝试保持一一可审计。

RunStore 不恢复 worker

RunStore 恢复的是历史账本和状态记录，不恢复执行现场。store_only RunRecord 用于查询和 orphan 修正，不用于接管正在执行的 task。

状态优先于观察通道

checkpoint 是下一次执行的事实来源，SSE 是观察通道。断连、重连或流式事件缺失不应该改变 ThreadState 的恢复语义。

需要警惕的实现风险

checkpoint 与 event 的观察差异

agent.astream() 产出 chunk 前，LangGraph 可能已经推进状态并写 checkpoint。如果此时停止 publish，前端看到的过程可能落后于下一次 Run 读取到的状态。

cancel 是协作式取消

run_mgr.cancel() 可以先把 Run 标为 interrupted，但外部 IO、工具调用或模型请求未必立即停止。副作用工具需要幂等或补偿设计。

rollback 需要屏障

如果新 Run 在旧 Run 完成 rollback 前读取 checkpoint，就可能读到中间状态。更严格的抢占需要“旧 Run 终止并完成 checkpoint 边界后，新 Run 才启动”。

rollback 不是事务回滚

rollback 恢复的是 pre-run checkpoint，不能撤销已经发出的 SSE、已经写入的 RunEventStore 事件、ThreadMeta 展示元数据，也不能撤销工具调用、文件写入或外部 API 请求。

外部副作用需要幂等

失败显式化和新 Run 重试会让同一用户意图再次触发工具。会写外部系统的工具应设计幂等键、去重记录或补偿动作，否则用户重跑可能造成重复提交。

多节点需要执行权约束

如果部署层不保证 sticky routing 或单 worker，同一 Thread 可能被多个节点同时执行。此时需要全局 thread 锁或 lease，否则会出现多个 checkpoint writer。

因此，当前一致性边界的重点不是“所有通道完全同步”，而是让每个事实层的职责保持稳定：checkpoint 负责继续执行，RunStore 负责 Run 生命周期，RunEventStore 负责历史事件投影，SSE 负责在线观察，外部副作用则需要由工具自身提供幂等或补偿能力。

分布式扩展取舍

在当前 DeerFlow 设计下，分布式扩展不是理解 Run Runtime 的前置条件。单 worker 或 sticky routing 已经能自洽地表达 Run 创建、流式输出、取消、回滚和失败显式化。分布式讨论的价值在于说明：如果未来目标变成长任务、强 SLA、多节点任意接管，就必须新增哪些运行时协议。

当前文章必须讨论的部分

需要说明 DeerFlow 当前不恢复 worker 现场、不跨节点迁移 task，并通过失败终态 + 新 Run + checkpoint 继续来保持模型清晰。

可以作为扩展方向的部分

分布式锁、worker lease、跨节点 stream、持久取消意图等不是当前设计必备能力。它们属于从交互式 Agent 走向 durable runtime 时才需要引入的增强。

如果要让同一个 Thread 的多次 Run 分散到不同节点，并支持主动打断、排队和 join stream，就需要把“执行权”从进程内对象提升为共享运行时协议。

能力 当前实现 选择分布式后需要补充
Thread 执行互斥 RunManager 本进程锁和本地 inflight 索引。 数据库行锁、Redis lock、lease 或 fencing token，保证同一 Thread 同时只有一个 checkpoint writer。
取消意图 写入本地 abort_event 并 cancel 本地 task。 持久化 cancel_requested、cancel_action，由目标 worker 心跳或控制通道感知。
Worker ownership RunRecord 的 task 指向本进程 task。 RunRecord 记录 worker_id、lease 过期时间和心跳，支持 orphan recovery。
抢占屏障 创建新 Run 时会请求取消旧 Run，但不天然等待 rollback 完成。 引入 cancelling、rolling_back、queued 等中间态，旧 Run terminal 后再启动新 Run。
实时事件 MemoryStreamBridge 进程内 buffer。 Redis Streams、消息队列或数据库事件流，让任意节点都能 join / replay。

分布式一致性的核心不是让每个节点都能读 checkpoint，而是保证同一 Thread 的执行权、取消意图、checkpoint 写入和新 Run 启动顺序有一个全局可验证的状态机。

源码阅读路径

阅读这条链路时，可以从 Gateway 入口开始，再进入 runtime 组件。重点不是逐行背代码，而是观察每个组件在哪个状态边界上写入事实。

  • backend/app/gateway/routers/thread_runs.py：Run API、stream、wait、cancel、join 的 HTTP 入口。
  • backend/app/gateway/services.py：start_run()、输入归一化、config/context 合并、worker task 启动、SSE consumer。

  • backend/packages/harness/deerflow/runtime/runs/worker.py：run_agent()、pre-run checkpoint snapshot、agent.astream、interrupt/rollback、finally 收尾。
  • backend/packages/harness/deerflow/runtime/stream_bridge/memory.py：当前 StreamBridge 的进程内事件缓冲、replay、heartbeat 和 cleanup。
  • backend/packages/harness/deerflow/runtime/checkpointer/async_provider.py：memory、sqlite、postgres checkpointer 的创建方式。
  • backend/packages/harness/deerflow/runtime/events/store/：持久化 messages、run events、token 和 audit 信息的事件存储。
