DeerFlow 研究：数据层设计、API 语义与展示投影

从 messages 看 DeerFlow 的数据分层

DeerFlow 中的 messages 不是一个单一字段，而是一组分布在不同层次上的数据语义。它既可能表示 agent 下一轮执行要读取的对话状态，也可能表示运行过程中的展示事件、SSE 在线片段，或者前端合并后的 UI 状态。因此，同一个 thread_id 下，不同接口和不同运行阶段看到的 messages 并不一定相同。

例如，/threads/{thread_id}/state 返回的是 checkpoint 中的 ThreadState.messages，它服务于后续执行；/threads/{thread_id}/messages 返回的是 RunEventStore 中的消息事件投影，服务于分页展示、反馈和审计；SSE 里的 event: messages 则是当前 Run 的实时传输片段，服务于在线观察。它们共享“messages”这个名字，但回答的是不同问题。

名字 所在位置 真实语义
ThreadState.messages Checkpointer 的 channel_values 下一轮 agent 继续执行时读取的权威对话状态。
/threads/{id}/messages RunEventStore 跨 Run 的可展示消息事件投影，适合分页、反馈和审计视角。
SSE messages event StreamBridge 在线流 当前 Run 执行过程中的实时消息片段，由 SDK 合并。
前端 thread.messages LangGraph SDK + DeerFlow hook 历史、实时流和乐观消息合并后的 UI 输入，不是后端事实源。

判断 DeerFlow 的 messages 时，首先要区分它是执行事实、事件投影、实时流，还是前端合成结果。

这个分层视角能解释很多看似矛盾的现象：一个 Run 已经成功、message_count 也有值，但 /runs/{run_id}/messages 仍然可能为空；刷新页面可以恢复历史会话，却不一定能重新接入正在运行的 SSE stream；换浏览器后可以找回 thread，却可能看不到完整的运行过程细节。根因都在于事实源、传输通道和展示投影并不是同一层数据。

先建立五层数据模型

要把 DeerFlow 看清楚，最好先暂时放下“聊天应用”的直觉。普通聊天系统常把问题简化成：用户发一条消息，服务端存一条 assistant 回复，前端把消息列表渲染出来。但 DeerFlow 不是只生成回复，它会规划、调用工具、运行子任务、产生 artifact、记录 token、支持中断和重连，还要让下一轮 agent 能接着执行。

因此 DeerFlow 的数据设计更接近一个运行时系统：不同数据层回答不同问题，不能混在一个“messages 表”里。

层次 核心问题 设计取舍
状态层 下一轮 agent 应该从什么上下文继续？ 必须稳定、可恢复、面向执行语义；对应 Checkpointer 和 ThreadState。
账本层 一次 Run 是否存在、是否完成、消耗了多少资源？ 必须可查询、可聚合、适合列表和统计；对应 RunStore。
事件层 运行过程中发生过哪些可解释、可审计、可展示的细节？ 高频、体积大、可能敏感，可按部署需要独立持久化；对应 RunEventStore。
传输层 浏览器如何在线观察一个正在运行的 Run？ 追求低延迟和顺序消费，不承担完整历史存储；对应 StreamBridge 和 SSE。
投影层 用户界面应该把这些数据组织成什么工作台体验？ 服务于展示、交互和局部反馈，不应反过来成为后端事实源；对应 SDK state 和 DeerFlow hooks。

这五层模型是理解全文的主线：状态用于继续执行，账本用于确认生命周期，事件用于解释过程，传输用于实时观察，投影用于组织用户体验。

后面讨论 API、SSE 和前端时，都可以回到这个模型判断：一个接口读到的空数组，可能只是某个投影层为空；一个 SSE 断开，也不代表状态层丢失；一个 Run 已经成功，也不保证事件层被持久化。

数据事实源地图

有了五层模型，再看 DeerFlow 的存储结构就不容易混淆。系统里几乎所有数据都围绕 thread_id 和 run_id 组织，但这两个 ID 只是关联键，不代表所有数据都在同一个地方，也不代表所有接口都在读同一份记录。

thread_id 更像一条长期会话线索：它串起 checkpoint、thread 列表、历史 Run 和 UI 工作台。run_id 则是一段具体执行的账本键：它标识一次 agent 运行、一次 SSE stream、一次 token 统计和一批过程事件。下面这张图先给出整体地图，后文再沿着一次 Run 的写入路径展开。

事实源 负责回答的问题 丢失后的影响
Checkpointer 这个 thread 下一轮从哪里继续？当前 messages、todos、artifacts 是什么？ 会话连续性受影响，是执行状态的核心事实源。
RunStore 这个 Run 是否成功、什么时候创建、用了多少 token、有多少消息？ 运行账本、统计和列表受影响，但不直接决定下一轮模型上下文。
RunEventStore 这次 Run 过程中发生了哪些 message、tool、trace、middleware 事件？ 历史消息投影、审计、事件调试受影响；不等于 checkpoint 丢失。
StreamBridge 当前浏览器如何实时看到 Run 事件？断线后还能从哪条 SSE event 继续？ 实时 join / replay 受影响，但已写入的 checkpoint 和 run 账本仍可存在。
ThreadMetaStore 侧边栏怎么列出 thread，title、status、updated_at 显示什么？ 会话列表受影响，但不等于 thread state 消失。
Memory 长期用户偏好、历史摘要、facts 如何注入到后续对话？ 个性化和长期记忆受影响，不影响单个 thread 的 checkpoint。

这张地图也暗含一个设计原则：越靠近“继续执行”的数据，越应该稳定持久；越靠近“观察过程”的数据，越容易被配置成可选持久化；越靠近“展示体验”的数据，越应该允许前端重新投影。这个原则解释了 DeerFlow 为什么没有把所有东西都绑定到一张统一消息表。

一次 Run 的数据写入路径

一次用户输入进入 DeerFlow 后，不是只写一个“assistant response”。它会先创建 Run 账本，再启动 worker。worker 执行时，一边把可恢复状态写进 Checkpointer，一边把过程细节写进 RunEventStore，同时通过 StreamBridge 把在线事件推给浏览器，最后再把统计和完成状态写回 RunStore。

这条写入路径体现了 DeerFlow 的核心取舍：恢复执行、解释过程、实时观察和最终展示是四件事。它们发生在同一次 Run 里，但不应该由同一个存储结构承担。下面用一个直接调用 Gateway 原生 /api 前缀的 curl 示例进入这条路径。

export DEERFLOW_API="http://localhost:2026/api"

thread_id=$(
  curl -s -X POST "$DEERFLOW_API/threads" \
    -H "Content-Type: application/json" \
    -d '{}' \
  | python -c 'import json,sys; print(json.load(sys.stdin)["thread_id"])'
)

curl -N -X POST "$DEERFLOW_API/threads/$thread_id/runs/stream" \
  -H "Accept: text/event-stream" \
  -H "Content-Type: application/json" \
  -d '{
    "assistant_id": "lead_agent",
    "input": {
      "messages": [
        {
          "type": "human",
          "content": [
            {
              "type": "text",
              "text": "请用三句话解释 DeerFlow 的 Run 生命周期"
            }
          ]
        }
      ]
    },
    "stream_mode": ["values", "messages-tuple"],
    "stream_subgraphs": true,
    "stream_resumable": true,
    "context": {
      "thread_id": "'"$thread_id"'",
      "is_plan_mode": false,
      "subagent_enabled": false
    },
    "on_disconnect": "continue",
    "multitask_strategy": "reject"
  }'

这条请求会打开一个 SSE 响应，但它背后更重要的是数据写入顺序：

阶段 写入或发布 作用
创建 Run RunManager / RunStore 写入 RunRecord 生成 run_id，记录 status、kwargs、thread_id、assistant_id。
返回 stream 响应 Content-Location: /api/threads/{thread_id}/runs/{run_id} 让 SDK 从响应头识别 canonical run resource 和 run_id。
worker 执行 LangGraph agent 读取并写入 Checkpointer 更新 ThreadState.messages、todo、artifact、title 等 channel values。
运行过程记录 RunJournal 写 RunEventStore 记录 human input、AI response、tool result、trace、middleware 等事件。
实时观察 StreamBridge 发布 metadata、messages、values、end Gateway SSE consumer 把事件转成浏览器可消费的 SSE frame。
收尾 RunStore 写 token usage、message_count、final status；ThreadMetaStore 同步 title/status 让 run 列表、thread 列表、token usage 和历史查询能展示最终结果。

前端 SDK 默认使用公开的 LangGraph-compatible 前缀 /api/langgraph。在本地 nginx 或 Next rewrites 下，/api/langgraph/threads/{id}/runs/stream 会被改写到 Gateway 原生的 /api/threads/{id}/runs/stream。因此看源码时常见的是 /api/threads，看浏览器网络请求时则可能看到 /api/langgraph/threads。

从数据视角看，runs/stream 不是一个单纯的“流式聊天接口”。它同时完成三件事：创建可查询的 Run 资源、启动会写 checkpoint 的后端执行、建立一个临时的实时观察通道。后续所有“重连”“join”“历史消息为空”的讨论，都来自这三件事被刻意拆开了。

API 设计：接口读哪个事实源

DeerFlow 的 API 名字很像一套统一资源模型，但它真正暴露的是多种读模型。只看 URL 很容易误判，所以更可靠的方式是问：这个接口回答的是执行状态、生命周期账本、事件投影，还是实时通道？

这种 API 设计不是偶然的。对用户来说，thread 和 run 是稳定的业务概念；对后端来说，状态、账本、事件和传输必须分开演进。因此 API 使用统一的资源路径组织入口，但在每个 endpoint 内部选择最适合回答该问题的事实源。它不是“一个资源对应一张表”的 CRUD 设计，而是围绕查询意图组织的读模型设计。

API 事实源 适合回答的问题
GET /api/threads/{thread_id}/state Checkpointer 当前 thread 的权威状态是什么？历史对话、todo、artifact 当前值是什么？
POST /api/threads/{thread_id}/history Checkpointer history 这个 thread 的 checkpoint 历史是什么？最新 checkpoint 里的 messages 是什么？
GET /api/threads/{thread_id}/runs RunStore 这个 thread 发生过哪些 Run？状态、时间、token usage、message count 是什么？
GET /api/threads/{thread_id}/runs/{run_id} RunStore 某个 Run 的生命周期账本是什么？是否成功？是否可取消？
GET /api/threads/{thread_id}/messages RunEventStore 跨 Run 的可展示 message event 投影是什么？哪些 message 可附加 feedback？
GET /api/threads/{thread_id}/runs/{run_id}/messages RunEventStore 某次 Run 的 message event 分页结果是什么？
GET /api/threads/{thread_id}/runs/{run_id}/events RunEventStore 某次 Run 的完整事件和 trace 是什么？
POST /api/threads/{thread_id}/runs/stream RunStore + Checkpointer + StreamBridge + RunEventStore 创建新 Run，并打开在线观察通道。
GET /api/threads/{thread_id}/runs/{run_id}/stream StreamBridge 重新接入一个 active Run 的 SSE stream，必要时带 Last-Event-ID。
POST /api/threads/{thread_id}/runs/{run_id}/stream?action=interrupt RunManager + StreamBridge 停止 Run，同时继续 stream 剩余 buffered events，让前端看到干净收尾。
GET /api/threads/{thread_id}/token-usage RunStore 聚合 thread 维度 token usage，可选择包含 active run 快照。

最常见误解：/threads/{thread_id}/messages 不是从 checkpoint 读取 ThreadState.messages，而是从 RunEventStore 读取 category="message" 的事件投影。查询真实历史对话时，应优先看 /threads/{thread_id}/state 或 /threads/{thread_id}/history。

这里有一个很实用的判断方法：如果你想知道“agent 下一轮会看到什么”，读 state 或 history；如果你想知道“这次运行是否成功、花了多少 token”，读 runs；如果你想知道“运行过程有哪些可展示/可审计事件”，读 messages 或 events；如果你想接入一个还在跑的任务，才去读 stream。

# 查询 thread 当前权威对话状态
curl -s "$DEERFLOW_API/threads/$THREAD_ID/state" \
  | jq '.values.messages'

# 查询 checkpoint history 中最新一条携带的 messages
curl -s -X POST "$DEERFLOW_API/threads/$THREAD_ID/history" \
  -H "Content-Type: application/json" \
  -d '{"limit": 1}' \
  | jq '.[0].values.messages'

SSE：实时通道不是持久状态

SSE 是 DeerFlow 在线观察一次 Run 的协议通道。worker 不直接写浏览器连接，而是把运行事件发布到 StreamBridge；Gateway 的 SSE consumer 订阅 StreamBridge，再格式化成浏览器和 LangGraph SDK 可消费的 frame。

这里要先拆开三个常被混用的动作：POST /runs/stream 是创建一个新 Run 并立即观察它；GET /runs/{run_id}/join 或 GET /runs/{run_id}/stream 是接入一个已经存在且仍在当前 worker 上 active 的 Run；POST /runs/{run_id}/stream?action=interrupt 则是先取消或回滚，再继续把剩余 buffered events 流给前端，保证停止按钮后的 UI 能看到收尾状态。

id: 1790000000000-12
event: messages
data: {...}

id: 1790000000000-13
event: values
data: {...}

metadata

标记当前连接属于哪个 thread、哪个 run、哪些 stream mode。

messages

模型消息和工具消息的在线片段，最终由 SDK 合并进前端状态。

values

ThreadState channel values 快照，例如 todo、artifact、title 等。

end / error

结束或异常信号，驱动前端恢复 ready、刷新查询或进入错误路径。

每个 SSE frame 可以带 id。客户端断线后，如果带着 Last-Event-ID 重连，MemoryStreamBridge 会尝试从保留 buffer 中找到对应事件，并从下一条开始 replay。这里的 replay 只补在线观察通道中仍保留的事件，不等于恢复 Python task 的执行现场，也不等于读取完整历史消息。

on_disconnect=continue 解决的是另一个问题：浏览器 SSE 连接断开后，后端 worker 是否继续执行。设置为 continue 时，Run 可以继续写 checkpoint、RunStore 和 RunEventStore；设置为 cancel 时，断连会触发取消语义。它不负责保证任意新浏览器都能接入原来的内存 stream。

Join 的语义也是围绕 StreamBridge：客户端不创建新 Run，而是接入已有 active Run 的 SSE stream。当前 DeerFlow 支持 GET /runs/{run_id}/join 和 GET/POST /runs/{run_id}/stream 这类入口；但这些入口能否工作，取决于持有该 Run 的 worker 是否还在，并且本进程内的 StreamBridge buffer 是否还在。当前 MemoryStreamBridge 的 buffer 是有界内存队列，超过保留窗口或进程切换后，Last-Event-ID 只能从最早保留事件开始 replay，甚至无法再提供实时 join。

容易误解

以为 Last-Event-ID 是一种历史消息查询协议，或者以为知道 run_id 就一定能接回完整运行过程。

更准确的理解

Last-Event-ID 只定位 StreamBridge buffer 里的事件偏移；run_id 可以从 RunStore 找回运行账本，但实时 join 仍要求 active worker 和 stream buffer 存在。

SSE 是观察通道，不是最终事实源。继续执行看 Checkpointer，生命周期看 RunStore，过程审计看 RunEventStore，在线重连才看 StreamBridge。

存储配置设计

DeerFlow 没有把所有数据都塞进一个“会话表”。核心业务数据、过程事件、实时 buffer 和长期记忆分别配置，是因为它们的数据量、可靠性要求、隐私风险和部署成本不同。

配置 控制的数据 当前常见取值
database Checkpointer、RunStore、ThreadMetaStore、Feedback、Users、Channel connections 等核心数据。 memory、sqlite、postgres。当前示例配置为 sqlite。
run_events RunEventStore 中的 message event、trace、tool result、LLM request/response 等过程明细。 memory、db、jsonl。当前示例配置为 memory。
stream_bridge active Run 的 SSE buffer、heartbeat、Last-Event-ID replay 和 join stream。 当前主要是 memory；Redis stream bridge 仍属于后续方向。
memory 用户长期记忆、历史摘要、facts 和注入上下文。 默认文件存储，例如 memory.json 或 per-user memory file。
paths / sandbox / uploads 上传文件、artifact、sandbox 工作目录、输出文件。 文件系统和容器 volume，取决于部署路径。

其中 run_events 单独配置尤其重要。Run events 是观测和审计数据，不是恢复执行所必需的核心状态；它写入频率高、体积可能大，还可能包含 prompt、tool 输入输出、文件路径和中间 trace。因此 DeerFlow 允许部署方只持久化核心账本和 checkpoint，而不持久保存完整事件明细。

database:
  backend: sqlite
  sqlite_dir: .deer-flow/data

run_events:
  backend: memory
  max_trace_content: 10240
  track_token_usage: true

memory:
  enabled: true
  storage_path: memory.json

在标准本地启动方式下，.deer-flow/data 会按 Gateway 进程工作目录解析。若 Gateway 从 backend/ 启动，SQLite 文件通常是：

/Users/bytedance/Projects/deer-flow/backend/.deer-flow/data/deerflow.db

这也解释了为什么同一个系统里，有些数据重启后还在，有些数据会消失：它们并不共享同一套持久化配置。

设计哲学：把事实和投影拆开

DeerFlow 的数据设计可以理解成一种轻量的读写分离：写入路径服务于运行时正确性，读取路径服务于不同交互问题。它没有试图让一个“messages”概念覆盖所有场景，而是承认 agent 运行天然包含多种事实。

执行事实

Checkpointer 保存 agent 继续执行所需的状态。它关心的是“下一步怎么跑”，不是“用户界面如何排版”。

账本事实

RunStore 保存一次运行的存在性、状态和资源消耗。它让系统能回答“发生过什么运行”。

过程事实

RunEventStore 保存可解释的中间过程。它让系统能回答“为什么会得到这个结果”。

展示投影

前端把状态、事件和流合并成工作台。它服务于用户理解和操作，不反向定义后端事实。

这个拆分的好处是边界清楚。RunEventStore 可以因为成本或隐私原因配置为 memory，但不应该因此影响 checkpoint 恢复；SSE 可以因为进程重启而丢失 buffer，但不应该因此抹掉 RunStore 账本；前端可以过滤隐藏消息、合并乐观消息、重组 subagent 卡片，但不应该改变后端认为的 ThreadState.messages。

当前取舍：DeerFlow 仍把 ThreadState.messages 作为 checkpoint 状态的一部分。这符合 LangGraph agent 的执行模型，因为恢复执行时需要拿到一组可直接喂给模型和 middleware 的消息上下文；但它也带来明确代价：对话越长、tool result 越多、subagent 过程越复杂，checkpoint 就越容易膨胀。

从设计上看，可以把“完整历史对话”和“恢复执行上下文”继续拆开。一个更强的演进方向是引入独立 ConversationStore 或 MessageStore，让它成为用户可见历史 transcript 的事实源；checkpoint 只保留恢复执行所需的最小闭包，例如最近若干条 materialized messages、summary、必要 tool call/tool result 对、artifact/todo/sandbox 状态，以及指向历史消息区间的稳定引用。

不过，这不应该被简化成“checkpoint 只存指针”。纯指针 checkpoint 会把恢复执行变成跨 store hydrate：需要处理消息版本、事务一致性、回滚到旧 checkpoint、分支历史、删除保留期、权限隔离和历史消息被压缩后的可重放性。更稳妥的边界是：历史 transcript 可以独立成为事实源，但 checkpoint 仍必须保存足够的 materialized execution context，保证 agent 恢复时不依赖不稳定的展示投影。

它的代价也很真实：调试时不能只看一个接口，必须知道每个接口背后的事实源；多端同步也不是天然成立，尤其是 active stream 仍依赖当前 worker 和内存 StreamBridge。若要面向分布式服务强化这部分能力，需要把 StreamBridge 从进程内 buffer 推进到可共享的传输层，例如 Redis stream 或其他外部 pub/sub，并让 join 路径不再依赖单机内存。

DeerFlow 的设计不是把复杂性藏起来，而是把复杂性分层：状态负责正确性，事件负责可解释性，流负责实时性，投影负责体验。

典型现象解释

把事实源区分清楚以后，一些看起来奇怪的现象就不再矛盾。

现象一：Run 有 message_count，但 /messages 为空

message_count 来自 RunJournal 写入 RunStore 的统计快照；/threads/{id}/messages 和 /runs/{id}/messages 来自 RunEventStore。如果 run_events.backend=memory 且 Gateway 重启过，RunStore 仍有统计，RunEventStore 事件明细可能已经丢失。

正确查询历史对话

如果目标是“这个 thread 当前的历史对话是什么”，应查询 /threads/{id}/state 的 values.messages，或查询 /threads/{id}/history 的最新 checkpoint，而不是依赖 RunEventStore message projection。

现象二：换浏览器后无法恢复未完成 stream

浏览器本地保存的 reconnect metadata 可能丢失，且 active stream buffer 在当前 worker 的 StreamBridge 里。换环境后仍可以从 ThreadMetaStore 和 Checkpointer 找回历史 thread，但不一定能接回未完成的实时流。

正确理解多端同步

on_disconnect=continue 可以让 worker 继续执行并写 checkpoint、RunStore、RunEventStore；但“能继续执行”不等于“任意客户端都能实时 join 当前内存 stream”。实时 join 仍受 StreamBridge 和 worker 亲和性限制。

现象三：只靠浏览器缓存保存 run_id

前端可以从 Content-Location、SSE metadata 或 SDK 内部状态拿到当前 run_id，也可能把 reconnect metadata 留在浏览器环境里。但如果把它理解成唯一索引，换浏览器或清缓存后就会误判为“会话丢了”。

正确找回运行记录

已创建的 Run 应从 RunStore 查询，例如先列出 thread，再读 /threads/{thread_id}/runs 找到最近的 run_id。这能恢复运行账本和最终状态判断，但仍不保证能重新接入 active SSE stream。

现象四：/api 和 /api/langgraph 看起来重复

Gateway 原生 router 挂在 /api/*；前端 SDK 默认使用 LangGraph-compatible 前缀 /api/langgraph/*。nginx 或 Next rewrite 会把后者转成 Gateway 原生路径。

正确区分路径语义

写源码和 curl 调试时可以使用 /api/threads；观察浏览器和 SDK 时常看到 /api/langgraph/threads。它们对 runs / threads / streaming 最终落到同一批 Gateway router。

展示交互：从事实到工作台

前端不是后端事实源，但它是用户理解 DeerFlow 的主要界面。它的价值不在于原样渲染每条后端事件，而在于把 Checkpointer、RunStore、RunEventStore、StreamBridge 和文件系统的数据投影成一个可操作的任务工作台。

LangGraph SDK 消费 SSE，把在线事件合并成 thread.messages 和 thread.values；DeerFlow 再在外层处理乐观消息、历史分页、上传状态、token usage、stop、join 和 UI 分组。这个过程本质上是把多源事实降噪、排序、分组，并转换成用户能理解的交互状态。

消息列表并不直接渲染后端原始消息。DeerFlow 会过滤 summary、loop_warning、todo_reminder、todo_completion_reminder 等隐藏控制消息，也会把消息分成 human、assistant:processing、assistant、assistant:present-files、assistant:subagent、assistant:clarification 等 UI 组。

前端区域 主要数据来源 投影逻辑
消息列表 SDK 合并后的 messages + RunEventStore 历史补齐 过滤隐藏消息、合并乐观消息、按 tool/subagent/clarification 分组。
Todo 面板 thread.values.todos 把 ThreadState channel value 投影成任务清单。
Artifact 面板 thread.values.artifacts + artifact API 从文件事实源读取预览和下载，不只是一段 markdown 链接。
Subagent 卡片 task tool call + custom event 把子任务的运行状态、最新消息和结果组织成卡片。
停止按钮 SDK existing run stream + RunManager cancel 前端触发控制意图，真正的 interrupt / rollback 由后端运行时完成。

因此，前端体验稳定性的关键不是把 UI state 当成事实源，而是建立清晰的恢复优先级：权威对话和 todo 从 checkpoint 恢复，运行列表和 token usage 从 RunStore 恢复，过程消息从 RunEventStore 补齐，正在运行的增量才通过 active SSE stream 观察。

这也解释了为什么 DeerFlow 前端需要同时处理“历史读取”和“在线流”。如果只读 checkpoint，用户会缺少运行过程、反馈位置和可解释细节；如果只依赖 SSE，刷新或换浏览器就会脆弱；如果只读 RunEventStore，又无法保证拿到下一轮 agent 真正会使用的状态。一个稳定的展示交互，必须承认这些数据源各自有边界。

结论

DeerFlow 的数据设计不是一个单表会话模型，而是多个事实源协作：Checkpointer 负责继续执行，RunStore 负责生命周期账本，RunEventStore 负责过程审计和展示消息投影，StreamBridge 负责在线观察，ThreadMetaStore 负责会话列表，Memory 和文件系统负责长期记忆与产物。

从这个角度看，SSE、API 和前端 UI 都只是数据流上的不同投影。判断一个问题时，先问“这条数据的事实源是谁”，再问“这个接口只是读投影，还是读权威状态”。这比单独研究某个前端 hook 或某个 SSE event 更能解释 DeerFlow 的工程边界。
