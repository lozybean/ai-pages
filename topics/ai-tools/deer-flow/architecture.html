核心模型

DeerFlow 可以理解成一个部署在 Gateway 内部的 LangGraph Agent Runtime。外部看到的是兼容 LangGraph Platform 的 HTTP/SSE 接口；内部真正工作的，是 Gateway 路由、RunManager、StreamBridge、worker、Agent factory、中间件链、模型与工具系统、checkpointer 和记忆队列。这里的 LangGraph Server 更像一套可兼容的协议和工具入口；默认服务路径并不是把请求转发给一个独立的 LangGraph Server 进程。

这套结构的关键价值是把“对话”拆成了几个明确边界：HTTP 入口只负责协议和权限，Run 层负责生命周期和流式事件，Agent 层负责模型与工具推理，Middleware 层负责把线程上下文、文件空间、标题、token、记忆等横切能力挂到同一条执行链上。

组件地图

从组件关系看，DeerFlow 不是一个单纯的聊天接口，而是“Gateway + 嵌入式 Agent harness”。Nginx 和前端只是入口形态；核心运行时在 Gateway 进程内完成。

API 设计

DeerFlow 的 Gateway 不是单纯的反向代理，而是同时承载两类 API 面：一类是面向 LangGraph SDK / 前端 useStream 的兼容协议，负责 thread、run、stream、state、history 等运行时交互；另一类是 DeerFlow 产品能力 API，负责模型、MCP、skills、memory、uploads、artifacts、channels 等应用级能力。

LangGraph 兼容 API

默认部署里，外部常见入口是 /api/langgraph/*。这组路径会被入口层重写到 Gateway 的原生 /api/* router，由 Gateway 进程内的 embedded runtime 创建 run、连接 StreamBridge、驱动 Agent，并按 LangGraph 客户端可识别的事件形状输出。

API 组 主要职责 调用方视角
Assistants /api/assistants 兼容 LangGraph assistant 查询、graph 信息和 schema 信息。 客户端可以发现当前可运行的 graph 入口，以及输入、输出、config schema。
Threads /api/threads 负责创建、查询、更新、删除 thread，并读取 state、history 等会话状态。 前端把 thread 理解为一次可持续对话的容器，而不是单次请求。
Thread Runs /api/threads/{thread_id}/runs 负责在线程内创建 run、等待 run、取消 run、join run、读取 run 消息、事件和 token usage。 客户端可以围绕一个 thread 连续发起多轮 run，并把每轮输出归到同一个会话上下文里。
Runs /api/runs/stream、/api/runs/wait 等 stateless run 入口提供不强依赖既有 thread 的运行方式。 适合一次性执行、调试或由调用方自己管理上下文的场景。

DeerFlow 应用 API

应用 API 不追求被通用 LangGraph 客户端直接理解，主要给 DeerFlow 的前端设置页、管理界面、外部集成和 channel 内部命令使用。Agent runtime 通常不会在执行过程中再通过 HTTP 调用这些 API；它读取的是同一套底层配置、storage、缓存或运行时服务。也就是说，应用 API 是“管理和观测入口”，底层资源才是 Agent 真正消费的运行时依赖。

API 组 主要调用方 对运行时的影响
Models / MCP 前端模型配置页、MCP 管理页、外部管理工具。 写入或展示模型与工具配置；run 创建后由 runtime config 和 tool registry 消费。
Skills 前端 skill 管理页、skill 安装/编辑流程、Agent 自演进工具的外部管理面。 读写 skill storage，并刷新 prompt 注入缓存；Agent 构建 prompt 和 tool policy 时直接从 storage/config 读取 enabled skills。
Memory 前端记忆管理页、IM/channel 内部命令、导入/清理/人工修正流程。 读写 memory storage；Agent prompt 注入时通过 memory updater/storage 读取并格式化，不是调用 /api/memory。
Uploads / Artifacts 前端文件上传、产物预览/下载、channel 文件接入。 uploads 进入线程文件空间；artifacts 来自 Agent 工具产出，并被前端按文件或结果产物渲染。
Agents / Suggestions 前端 Agent 选择、配置页、输入框建议和快捷任务入口。 影响用户如何发起任务，以及 run 初始化时选择哪类 Agent 配置、提示词和能力面。
Channels / Auth / Feedback 登录初始化、IM 集成、channel worker、反馈入口和运维界面。 负责运行时之外的接入、权限、运行状态和体验闭环；channel 可以代表外部用户触发 run。

端到端运行链路

一次对话不是普通请求响应，而是一条跨越入口、运行时、Agent、事件队列和前端渲染的端到端链路。可以从三条线同时理解：组件之间如何接力，用户界面如何变化，以及一个流式 chunk 如何从 Agent 内部抵达浏览器。

组件流动

组件流动关注“请求被谁接住、交给谁、再由谁继续推进”。这条线能帮助定位问题边界：是入口代理、Gateway 适配、Run 生命周期、Agent 执行，还是 SSE 输出。

图里的实线是执行推进方向，虚线是流式结果返回方向。Gateway router 和 Run Service 是协议适配边界；RunManager 和 RunRecord 是生命周期边界；Worker 和 Agent Runtime 是执行边界；StreamBridge 把 Agent 事件生产和 SSE 消费拆开。

用户侧视角

用户侧看到的是另一条时间线：页面不关心内部哪个类在运行，而关心会话是否创建、输入框是否忙、回答是否开始出现、工具状态是否可见、最终是否恢复可输入。

用户看到的变化 背后的运行时动作 对应事件或状态
发送消息后，输入框进入 busy/streaming。 前端提交 run，请求里带上 human message、stream 配置、mode/context 等运行参数。 前端本地 optimistic message + run submit。
新会话 URL 和侧边栏项变成真实 thread。 Worker 启动后先发布 run/thread 元数据，前端记录真实身份。 metadata
回答开始出现，并逐步变长。 模型或工具链路持续产生 message chunk，SDK 合并成当前最新的 thread.messages。 messages
标题、TodoList、artifact 入口等外围状态变化。 图状态被 checkpoint/channel values 更新，前端把最新 ThreadState 合并到 thread.values。 values
工具步骤、子任务卡片、文件列表或澄清块出现。 前端根据 AIMessage.tool_calls、ToolMessage 和工具名做 message grouping。 messages + 前端分组逻辑
回答结束，输入框恢复 ready。 Worker 完成收尾，StreamBridge 发布结束信号，SDK 进入 finish 路径并刷新相关缓存。 end

chunk 侧视角

chunk 侧视角适合理解“打字机观感”从哪里来。前端通常不是拿完整答案做字符级 diff 再定时打印，而是不断收到新的 message/state 事件，SDK 合并成当前完整内容，React 按最新内容重渲染。

这张图的关键点是：Worker 和 SSE consumer 并不直接互相等待。Worker 只负责把 chunk 变成事件并发布；StreamBridge 保留 per-run 事件日志、唤醒消费者，并支持基于事件 id 的重连 replay；前端看到的“打字机”来自 SDK 合并后的内容不断变长，而不是 StreamBridge 主动拆字。

Run 生命周期

Run 层的核心职责是把“不确定的 Agent 执行”变成“可管理的后台任务”。它不负责模型怎么想，而负责让这个过程能被创建、取消、回滚、观察、持久化和流式消费。

阶段 内部动作 对系统的意义
创建 生成 RunRecord，绑定 thread、assistant、metadata、model_name、断连策略。 让一次执行有稳定身份，便于查询、取消和审计。
启动 状态置为 running，捕获 pre-run checkpoint，发布 metadata。 客户端能立即知道 run/thread 身份，系统也保留回滚基础。
执行 构造 runtime context、agent、checkpointer、store、callbacks，再进入 LangGraph stream。 把 HTTP 请求切换为可持续推进的图执行。
结束 根据执行结果设置 success/error/interrupted，刷新 journal，持久化 token 与完成信息。 让 run 结果和可观测数据落到统一位置。
收尾 同步标题到 thread display name，更新 thread 状态，发布 end，延迟清理 stream queue。 让 UI 列表、历史记录和长连接生命周期保持一致。

Run 状态轮转

从状态机看，pending、running 是活跃态；success、error、interrupted 是终态。失败或异常后可以在同一个 thread 上再次发起执行，但那会创建新的 RunRecord 和新的 run_id；旧 run 不会从 error 回到 pending 或 running。代码里的短重试主要用于持久化写入、模型 provider 等内部细节，不等同于 run 级别的原地重跑。

Run Runtime 组件

Run Runtime 不是一个单点模块，而是一组边界清楚的协作组件。它们把 Gateway 收到的一次 run 请求拆成四件事：创建可追踪记录、启动后台执行、把执行事件交给 SSE 消费端、在结束时同步状态和清理资源。

组件 位置 核心职责 边界判断
Run Service Gateway service 层 读取 Gateway 单例依赖，校验 thread 权限和模型 allowlist，处理 on_disconnect、context、metadata、stream_mode，创建 run 记录并启动后台任务。 它是 HTTP 协议到 runtime 的适配层，不直接执行模型，也不直接决定 LangGraph 节点如何流转。
RunManager runtime/runs 维护 RunRecord、run 状态、thread 维度索引、并发策略、取消/中断信号，以及可选的 RunStore 持久化。 它关心“run 是否存在、能否启动、现在是什么状态”，不关心 chunk 内容如何渲染。
RunRecord runtime/runs 承载一次执行的稳定身份和运行元数据，包括 run_id、thread_id、assistant、status、metadata、kwargs、task、abort event、model_name、user_id 等。 它是 run 生命周期的事实载体；HTTP response、取消、wait、stream 和后台 worker 都围绕它协作。
StreamBridge runtime/stream_bridge 提供 producer/consumer 队列接口：worker 通过 publish 写入事件，SSE consumer 通过 subscribe 读取事件，同时支持 heartbeat、end sentinel 和延迟 cleanup。 它只负责事件传递和长连接解耦，不理解 message、tool、artifact 的产品语义。
Worker runtime/runs/worker.py 后台执行主体。它设置 run 状态，捕获 pre-run checkpoint，发布 metadata，构造 runtime context、agent、checkpointer、store 和 journal，调用 agent.astream() 并把 LangGraph chunk 转成 SSE 事件。 它是“真正推动 Agent 往前跑”的地方；但前端连接、权限校验和 HTTP response 生命周期仍由 Gateway/Service 层处理。
SSE consumer Gateway service 层 订阅 StreamBridge，把内部 StreamEvent 格式化成 SSE frame，处理 heartbeat、end 和客户端断连。 它是事件消费者，不生成 Agent 内容；断连时只按 on_disconnect 策略决定取消或继续后台任务。

Run 协作链路

这条链路的核心是“创建”和“执行”分离：Run Service 负责把 HTTP 请求规范化成 RunRecord 并启动后台 task；Worker 负责推进 Agent；StreamBridge 负责把 chunk 转成可订阅事件；SSE consumer 只负责把事件送到浏览器。结束时再由 Worker 统一更新状态、刷新 journal、同步 thread 信息并发布 end。

Agent 管线

Agent Runtime 是 DeerFlow 最值得反复看的部分。它不是“系统提示 + 模型调用”这么简单，而是通过 middleware 把线程数据、上传文件、沙箱、工具错误、技能、总结、todo、token、标题、记忆、视觉输入、subagent 限流和澄清中断接入同一条执行链。

Middleware 切面列表

Lead Agent 的 middleware 不是无序插件，而是按运行时职责逐层挂载。下面按常见注册顺序展开：基础运行时 middleware 先处理 thread、sandbox、工具安全和错误；随后 lead-agent 专属 middleware 处理动态上下文、skills、摘要、todo、标题、记忆、视觉、延迟工具、subagent、循环检测、安全终止和澄清。

  1. 01. ToolOutputBudgetMiddleware

    启用条件：默认。

    主要作用：控制工具输出预算，避免过大的 tool result 继续污染后续模型上下文。

    • Wrap Model：在模型绑定前处理工具输出预算相关的上下文约束。
    • Wrap Tool：工具返回后压缩或裁剪结果，再交给后续消息链。

  2. 02. ThreadDataMiddleware

    启用条件：默认。

    主要作用：建立 thread workspace、uploads、outputs 等线程工作区，并把线程运行元数据放入 state。

    • Before Agent：在 Agent 进入模型循环前准备线程目录、虚拟路径和 thread_data。

  3. 03. UploadsMiddleware

    启用条件：Lead Agent 默认。

    主要作用：读取线程上传文件，生成可被模型理解的文件摘要、预览和引用上下文。

    • Before Agent：扫描当前 thread 的上传文件，并把上传文件信息合并进 state / 用户上下文。

  4. 04. SandboxMiddleware

    启用条件：默认。

    主要作用：懒初始化 sandbox，给工具调用提供隔离环境，并在 Agent 收尾时处理沙箱状态。

    • Before Agent：按需准备 sandbox_id、工作目录和工具运行环境。
    • Wrap Tool：在工具执行时注入 sandbox 上下文，让文件、命令、运行目录落在隔离空间内。
    • After Agent：在执行结束后做沙箱状态回写或清理。

  5. 05. DanglingToolCallMiddleware

    启用条件：默认。

    主要作用：修复历史消息里缺失的 ToolMessage，避免模型看到不完整的 tool call 对。

    • Wrap Model：在请求送入模型前补齐 dangling tool call 对应的占位工具消息。

  6. 06. LLMErrorHandlingMiddleware

    启用条件：默认。

    主要作用：包裹模型调用，把 provider 异常转换成可恢复或可上报的失败形态。

    • Wrap Model：捕获模型调用异常，生成 fallback/error 结果，避免异常直接撕裂整条执行链。

  7. 07. GuardrailMiddleware

    启用条件：guardrails 配置启用。

    主要作用：在工具执行前做策略评估，被拒绝的工具调用会变成错误 ToolMessage。

    • Wrap Tool：把 tool name、tool input、agent id 等组装成 guardrail request，决定放行或拦截。

  8. 08. SandboxAuditMiddleware

    启用条件：默认。

    主要作用：审计高风险 sandbox/bash 类工具调用，负责安全记录与拦截前置判断。

    • Wrap Tool：在命令类工具执行前分析命令风险，并记录审计结果。

  9. 09. ToolErrorHandlingMiddleware

    启用条件：默认。

    主要作用：捕获工具异常并转成错误 ToolMessage，同时给 task 工具补结构化 subagent 状态。

    • Wrap Tool：包裹真实工具执行，把异常、安全失败或 subagent 结束状态规范化成前端可识别的工具消息。

  10. 10. DynamicContextMiddleware

    启用条件：Lead Agent 默认。

    主要作用：把当前日期、可选 memory 等动态上下文注入隐藏 reminder，保持静态 system prompt 稳定。

    • Before Agent：在 Agent 开始前读取运行时上下文和 memory，并把 reminder 写入消息附近。

  11. 11. SkillActivationMiddleware

    启用条件：Lead Agent 默认。

    主要作用：处理用户显式 /skill-name 激活，在模型调用前注入对应 skill 内容。

    • Wrap Model：检查用户消息里的 slash skill 激活，读取 skill 文件并改写模型请求消息。

  12. 12. DeerFlowSummarizationMiddleware

    启用条件：summarization 配置启用。

    主要作用：在模型调用前压缩长历史，并可在摘要前触发 memory flush hook。

    • Before Model：判断上下文是否达到摘要触发条件，生成摘要并裁剪历史消息。

  13. 13. TodoMiddleware

    启用条件：plan mode 启用。

    主要作用：提供 todo 工具、维护待办状态，并在未完成任务时把控制权跳回模型继续执行。

    • Before Agent：清理上一轮遗留的完成提醒，准备本轮 todo 运行上下文。
    • Before Model：把 todo 相关系统提示和状态整理给模型。
    • Wrap Model：向模型请求追加隐藏的 todo 完成提醒。
    • After Model：如果模型想结束但仍有未完成 todo，插入提醒并跳回 model。
    • After Agent：清理当前 run 的 todo completion reminder。

  14. 14. TokenUsageMiddleware

    启用条件：token usage 配置启用。

    主要作用：在模型返回后统计 token，并给 journal / run completion 写入调用归因。

    • After Model：读取模型响应里的 usage metadata，按 lead agent、subagent、middleware 归因。

  15. 15. TitleMiddleware

    启用条件：Lead Agent 默认。

    主要作用：在首轮有效对话后生成 thread 标题，最终同步到 thread display name。

    • After Model：在模型产出后判断是否需要生成标题，并把结果写入 state。

  16. 16. MemoryMiddleware

    启用条件：Lead Agent 默认挂载，按 memory 配置生效。

    主要作用：Agent 结束后把对话放入异步记忆更新队列；记忆注入本身由动态上下文阶段读取。

    • After Agent：提取本轮对话并排入 memory update 队列，不阻塞用户侧响应。

  17. 17. ViewImageMiddleware

    启用条件：模型支持 vision 时启用。

    主要作用：把用户已查看图片整理成模型可见的视觉输入上下文。

    • Before Model：把 viewed_images 中的图片内容或描述注入模型请求。

  18. 18. DeferredToolFilterMiddleware

    启用条件：tool_search / deferred MCP 工具存在。

    主要作用：隐藏未 promoted 的工具 schema，并阻止模型调用尚未通过 tool_search 暴露的工具。

    • Wrap Model：从 request.tools 中过滤未 promoted 的 deferred tools。
    • Wrap Tool：如果模型直接调用仍处于 deferred 状态的工具，返回错误工具消息。

  19. 19. SubagentLimitMiddleware

    启用条件：subagent_enabled 启用。

    主要作用：限制并发 task 工具调用数量，避免一次模型输出派发过多 subagent。

    • After Model：检查模型输出的 tool calls，裁剪或改写超过并发上限的 task 调用。

  20. 20. LoopDetectionMiddleware

    启用条件：loop detection 配置启用。

    主要作用：检测重复工具调用或无效循环，必要时注入提醒并引导模型跳出循环。

    • Before Agent：清理其他 run 的遗留循环警告。
    • Wrap Model：把上一步检测到的循环警告追加到本次模型请求。
    • After Model：分析模型输出和工具调用模式，判断是否出现循环。
    • After Agent：清理当前 run 的 pending warnings。

  21. 21. SafetyFinishReasonMiddleware

    启用条件：safety finish 配置启用。

    主要作用：识别 provider safety termination，清理不应继续执行的 tool calls。

    • After Model：检查 finish reason / safety signal，必要时移除工具调用并注入安全说明。

  22. 22. ClarificationMiddleware

    启用条件：总是最后挂载。

    主要作用：拦截澄清工具调用，把 ask-clarification 转成可中断、可恢复的用户确认流程。

    • Wrap Tool：识别澄清工具，触发 interrupt / resume 语义，而不是把它当普通工具结果处理。

读这组列表时要注意两个方向：注册顺序影响同一切面的包裹关系；hook 切面决定它能改什么。比如 before_agent 更适合准备 state 和工作区，wrap_model_call 更适合改模型可见 messages/tools，after_model 更适合检查模型输出和决定是否跳回模型，wrap_tool_call 则是工具安全、错误转换和执行拦截的关键位置。

流式事件与状态同步

DeerFlow 的流式输出可以理解为“worker 把 LangGraph stream mode 翻译成前端可消费事件”。前端并不是逐条把原始 SSE frame 画到屏幕上，而是通过 LangGraph SDK 的 useStream 把事件合并成 thread.messages、thread.values、loading 状态和回调事件，然后由消息分组逻辑决定具体 UI 形态。

SSE 事件 进入前端后的状态 通常渲染成什么
metadata useStream.onCreated(meta) 拿到 thread_id 和 run_id，触发 onStart，同时把线程放进 sidebar/search cache，状态标成 busy。 新会话 URL 被替换成真实 thread 路径；标题区域先显示默认新会话标题；输入框进入 streaming/busy 状态。
values 合并到 thread.values 和 thread.messages。它是当前 DeerFlow ThreadState channel values 的最新快照，可能包含 messages、title、todos、artifacts、thread_data、sandbox、uploaded_files、viewed_images、promoted 等。 消息列表按最新 state 重排；title 更新侧边栏和顶部标题；todos 触发输入框上方的 TodoList；artifacts 影响右侧产物入口；其他字段主要服务工具、中间件和后续模型调用。
messages 客户端请求的 messages-tuple 在 worker 内部映射到 LangGraph messages mode，再由 SDK 合并成单条或多条 AIMessage / ToolMessage。 前端再按 message role、tool call 名称、tool result 和隐藏标记分组。普通回答、thinking、工具步骤、subagent、文件展示和澄清消息都来自这一类事件。
events / LangChain callbacks 前端的 onLangChainEvent 会监听如 on_tool_end 这类事件；onCustomEvent 处理 task_running、llm_retry 等自定义事件。 tool end 可触发 artifact 或相关 query 刷新；task_running 更新子任务卡片最新消息；llm_retry 通常弹 toast，而不是插入聊天消息。
error worker 捕获异常或 LLM fallback 失败时发布；SDK 进入 onError。 清空乐观消息，输入区退出 streaming，页面 toast 展示错误；token usage 等相关 query 会按需刷新。
end worker finally 阶段发布；SDK 进入 finish 路径。 输入框恢复 ready；sidebar/thread search cache 刷新；token usage query 刷新；如果页面不在前台，前端会用最终回答摘要触发系统通知。

因此，工具、技能和子任务并不是靠单独的“tool event”或“skill event”直接画 UI。多数可见内容最终都落在 messages 或 values 里：AIMessage.tool_calls 决定显示哪个处理步骤，ToolMessage 补上结果，values 里的 title/todos/artifacts 更新页面外围状态。真正的原始事件层更偏运行时协议，前端用户看到的是经过 state 合并和 message grouping 后的产品形态。

messages 类型

messages 是最容易误解的一类输出：它不是只有“模型回答文本”，而是 LangChain message、tool call 和 tool result 的组合。DeerFlow 前端会先把原始消息过滤和归组，再决定显示成普通回答、处理中步骤、子任务卡片、文件列表或澄清块。

消息形态 识别依据 渲染动作
普通 AI 文本 AIMessage 有可见 content，且不是隐藏控制消息。 显示为 assistant markdown 气泡。流式中的最后一条 assistant 气泡会显示 loading/streaming 状态。
Thinking 内容 reasoning_content、<think> 或前端可识别的 reasoning 片段。 进入 Chain-of-Thought 的 Thinking 区，而不是作为普通正文直接混入回答。
普通工具调用 AIMessage.tool_calls 中出现 web_search、read_file、bash、write_file、write_todos 等工具名；随后同 id 的 ToolMessage 带回结果。 归入 assistant:processing 分组，渲染为 Chain-of-Thought 步骤。搜索显示结果，读文件显示路径，bash 显示命令代码块，文件写入类工具可打开 artifact 预览。
Subagent 派发 AIMessage.tool_calls 中工具名为 task，后续 ToolMessage.tool_call_id 对应这个 task id。 进入 assistant:subagent 分组，创建或更新 SubtaskCard，展示子任务执行中、完成或失败状态。
文件展示 模型调用 present_files，文件路径出现在 tool args 中。 进入 assistant:present-files 分组。前端先显示 assistant 文本，再用 ArtifactFileList 展示文件列表。
澄清消息 ask_clarification 的 ToolMessage 被识别为澄清消息。 保留在 processing group 中维持工具关联，同时打开 assistant:clarification 分组，把澄清内容作为醒目的 markdown 消息展示给用户。
Skill 激活后的输出 严格的 /skill-name ... 文本由 SkillActivationMiddleware 在模型前解析，插入隐藏 skill reminder，并通过 RunJournal 记录审计事件。 Skill 激活本身通常不是可见聊天气泡；用户看到的是后续模型输出、工具调用或产物。如果 skill 调用搜索、shell 或文件工具，仍按对应 message 类型渲染。

ThreadState 字段

values 事件里最核心的数据就是当前图的 channel values。对 DeerFlow lead agent 来说，这个 state 不是裸的 LangGraph 状态，而是 harness 对 LangChain AgentState 的细化：ThreadState。它保留了 agent 对话所需的 messages，同时加入 DeerFlow 自己的文件空间、沙箱、标题、产物、todo、上传文件、图片上下文和 deferred tool promotion 状态。

字段 含义 主要作用
messages 继承自 LangChain AgentState 的消息 channel，包含 human、ai、tool 等消息，也可能包含隐藏控制消息。 这是模型上下文和前端消息列表的主来源。前端会过滤隐藏消息，再按 human、assistant、processing、subagent、present-files、clarification 等分组渲染。
sandbox { sandbox_id }，记录当前 thread 绑定的沙箱实例。 让 shell、文件工具和沙箱生命周期能在同一轮或多轮执行中复用同一个隔离环境。reducer 只接受幂等写入；如果同一 thread 出现不同 sandbox id，会直接报错。
thread_data workspace_path、uploads_path、outputs_path 三类 per-thread 路径。 给工具层提供线程级文件空间。读写文件、上传文件、产物导出、路径脱敏和虚拟路径映射都依赖它。
title 线程标题，通常由 TitleMiddleware 在首轮完整对话后生成。 同步到 thread display name，前端顶部标题和侧边栏会用它替换默认新会话标题。
artifacts agent 展示或生成的产物路径列表。 前端 artifact 入口和文件列表用它展示可打开产物。reducer 会合并并去重，保持路径出现顺序。
todos plan mode 下的任务列表，由 write_todos 工具更新。 前端输入框上方的 TodoList 来自这里。reducer 语义是“最后一次非 None 更新获胜”，因此空列表也是一次明确清空。
uploaded_files 用户上传文件的元数据列表，例如文件名、大小、虚拟路径和上传状态。 UploadsMiddleware 会把它格式化为隐藏的 <uploaded_files> 上下文，让模型知道本轮可用文件。
viewed_images 按图片路径索引的图片内容缓存，值包含 base64 和 mime_type。 view_image 工具把读到的图片写入这里，ViewImageMiddleware 再在模型调用前注入视觉输入。空 dict 有特殊含义：清空已处理图片。
promoted { catalog_hash, names }，记录 deferred tool 中已经被提升可见的工具名。 配合 tool_search 和 DeferredToolFilterMiddleware 使用。catalog hash 变化时会整体替换，避免旧工具名在新工具目录下误暴露。

运行一致性

运行一致性关注的是：一次 run 从创建到结束，哪些状态必须保持可查询、可恢复、可取消、可审计。DeerFlow 没有把这些职责压到一个对象里，而是拆成 RunRecord、RunStore、checkpointer、ThreadMetaStore 和 StreamBridge 几层。

一致性对象 负责什么 实现细节
RunRecord 一次执行的内存事实载体。 创建时生成新的 run_id，初始为 pending；Service 启动后台 task 后回填到 record；取消时通过 abort_event 和 abort_action 传递 interrupt / rollback 意图。
RunManager run 状态机、thread 内并发策略和索引。 create_or_reject 会在同一把锁内检查 inflight run 并创建新 run；reject 阻止同线程并发，interrupt / rollback 会先取消旧 run 再创建新 run。
RunStore run 元数据、状态、token 和 completion 的持久化。 状态更新会写入 store；短暂数据库异常有 bounded retry。Gateway 启动时会把没有本地 worker 持有的历史 pending/running run 标记为 error，避免 UI 永远显示运行中。
LangGraph checkpointer 保存 ThreadState 的 checkpoint、history 和 channel values。 worker 执行前捕获 pre-run checkpoint；rollback 时可恢复到运行前状态；/wait 只有观察到 end 后才序列化最终 checkpoint，避免把半成品 state 当成成功结果返回。
ThreadMetaStore thread 列表、owner、display name 和运行状态。 run 创建时 upsert thread meta 并置为 running；worker finally 中从 checkpoint 读 title 同步 display name，并把成功 run 对应的 thread 状态置回 idle，失败或中断则写入对应状态。
StreamBridge 事件生产与 SSE 消费之间的一致性缓冲。 worker 发布 metadata、values、messages、error、end；consumer 支持 heartbeat、Last-Event-ID 重连 replay 和 end sentinel；run 结束后延迟 cleanup，给前端收尾和短暂重连留窗口。

运行一致性的核心不是让所有状态“同时写完”，而是让每一层有清晰的事实来源：run 是否活着看 RunManager / RunStore，thread 当前内容看 checkpointer，列表与归属看 ThreadMetaStore，浏览器流式消费看 StreamBridge。

关停路径也服务这个目标。Gateway shutdown 时会先 drain in-flight runs，再关闭 checkpointer 资源，避免后台 graph 还在写 checkpoint 时连接池已经被释放。对于无法在窗口内完成的 run，系统会记录风险并在下次启动时通过 orphaned inflight recovery 把状态收敛到 error。

记忆管理

记忆管理解决的是长期上下文如何产生、保存和再次注入。它不等同于 checkpoint：checkpoint 保存当前 thread 的运行状态，memory 保存跨 run、跨 thread 仍可能有价值的用户偏好、事实和历史摘要。

阶段 参与组件 实现细节
注入 DynamicContextMiddleware、memory storage、format_memory_for_injection 模型调用前读取当前用户 / agent 的 memory，按 max_injection_tokens 裁剪并包进隐藏 <memory> 上下文。token 估算可用 tiktoken 或无网络依赖的 char 策略。
采集 MemoryMiddleware.after_agent Agent 结束后从 state 里取 messages，只保留用户输入和最终 assistant response，忽略工具过程；同时检测 correction / reinforcement 信号，并在请求上下文还存在时捕获 user_id。
入队 MemoryUpdateQueue 队列按 thread_id + user_id + agent_name 去重 debounce。窗口内多次更新会合并为最新 context，避免每轮小消息都触发一次记忆模型调用。
提炼 MemoryUpdater、memory update prompt、LLM 后台线程处理队列，使用同步 model.invoke() 路径调用 memory model，避免复用 lead agent 的 async HTTP client 造成跨 event loop 连接问题。模型输出 JSON 后再做解析和规则化。
清洗 format_conversation_for_memory、updater 规则 上传文件标签会从 human message 中剥离，避免把临时文件路径写入长期记忆；过长消息会截断；事实会按 confidence、最大数量和更新规则合并。
保存 FileMemoryStorage 或自定义 storage class 默认按用户隔离到 {base_dir}/users/{user_id}/memory.json，也支持 per-agent memory。文件存储按 mtime 缓存读取，保存时写临时文件再 replace，更新 lastUpdated。
管理 /api/memory、前端记忆管理页、channel 命令 API 用于查看、reload、清空、导入和人工维护 memory facts。Agent runtime 通常不通过 HTTP 调这个 API，而是直接读写同一套 memory storage / updater。

为什么异步更新

记忆提炼需要额外模型调用，且失败不应该影响当前回答。放到 debounce 队列后，主 run 可以先完成，长期记忆以 best-effort 方式收敛。

为什么不写入所有消息

工具调用、上传文件路径、临时错误和中间推理容易污染长期记忆。DeerFlow 只保留用户输入和最终回答，再由 memory prompt 决定哪些内容值得沉淀。
