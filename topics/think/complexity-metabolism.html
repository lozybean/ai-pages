思考专题｜AI Coding 与工程吸收能力

复杂度代谢

AI Coding 时代，工程的瓶颈正在从生成效率转向吸收能力。真正稀缺的不是产出，而是把产出转化为可靠、可维护、可删除、可复用的软件能力。

AI Coding 复杂度代谢 工程治理 吸收能力 2026-05-18

AI Coding 的最大变化，不只是代码写得更快，而是生成成本突然降到了接近零。代码、文档、测试、脚本、方案、Agent 链路都可以快速生成，但人的判断、团队的吸收、系统的维护，并没有同步变快。

这会制造一个新的工程问题：产出变得太容易，而吸收、验证、删除和沉淀仍然很难。

过去，写代码本身很贵。一个功能要设计、实现、调试、测试、上线，生产成本天然会限制产出规模。现在，AI 可以在很短时间内生成大量代码和方案，让“增加”变得前所未有地便宜。

但软件系统的长期成本，通常不来自第一次写出来，而来自后续理解、维护、排障、扩展、删除和承担责任。AI 降低了生成成本，却不自动降低这些长期成本。

AI Coding 时代，真正稀缺的不是生成能力，而是把生成物转化为可靠能力的代谢能力。

一、AI 让生成进入过剩时代

AI 产出很像现代高热量食物。

高热量食物本身不是问题，问题是摄入变得太容易，而人的奖励系统和代谢机制还停留在过去。短期满足很强，长期代价很隐蔽。

AI Coding 也是类似。代码不是有毒，文档不是有毒，Agent 不是有毒，token 也不是有毒。问题是它们变得太容易生成，而人类和组织仍然容易把“有产出痕迹”误认为“有真实价值”。

AI 让什么变便宜短期感受长期风险
代码生成功能很快出现重复实现、隐含假设、维护成本上升
文档生成沉淀看起来充分长而不准、没人维护、真实知识更分散
测试生成覆盖看起来变多测试不贴近真实风险，形成虚假安全感
方案生成思路显得完整没有失败条件，无法区分好坏方案
Agent 链路生成流程显得智能链路过长、责任模糊、调试困难

真正危险的不是 AI 不会生成，而是生成太顺滑以后，团队失去了停下来判断的动力。

AI 让产出突然现代化了，但我们的判断和代谢机制还停留在产出稀缺时代。

二、瓶颈从生产能力转向吸收能力

AI 之后，问题不再只是“能不能写出来”。更关键的问题变成：

  • 团队能不能理解它？
  • 系统能不能接住它？
  • 测试能不能覆盖它？
  • 线上能不能观测它？
  • 维护者能不能负责它？
  • 未来能不能安全删除它？
  • 它能不能变成可复用资产，而不是一次性堆积？

这些问题共同指向一个概念：吸收能力。

吸收不是“把代码合进去”。真正的吸收是：AI 产出进入系统以后，被理解、被验证、被维护、被复用，并且在未来必要时可以被修改或删除。

AI Coding 的有效产出 = 被吸收的可靠能力 - 新增复杂度负债

如果一个 AI 生成物没有被理解，没有 owner，没有测试，没有复用，没有删除条件，它并没有真正被吸收。它只是进入了仓库，变成未来某个人需要理解的负担。

未被吸收的产出，不是库存，而是负债。

三、代码量和 token 是成本，不是成果

AI Coding 时代，最容易出错的是目标函数。

如果用代码量衡量工作量，系统会奖励多写少删；如果用 token 消耗衡量工作量，系统会奖励多调用、长上下文、复杂链路；如果用 Agent 节点数量衡量智能化程度，系统会奖励流程膨胀。

错误指标会鼓励什么会惩罚什么
代码量多写、重复写、少删除复用、简化、删除历史复杂度
token 消耗多调用、长链路、上下文堆积压缩、沉淀、少问、先思考
PR 数量频繁改动、可见动作少改、稳定、避免错误复杂度
Agent 节点数流程复杂化、自动化表演简单规则、人工判断、短链路
文档长度堆解释、堆背景清晰压缩、可执行结论

代码量可以观察复杂度变化，token 可以观察资源消耗，PR 数可以观察活动频率。但它们都不是价值本身。

错误目标

价值 = 代码量 + token 消耗 + PR 数 + Agent 节点数

更好的目标

价值 = 可靠结果 - 复杂度成本 - 维护成本 - 资源消耗 - 风险暴露

把成本当成果，系统就会奖励自己变胖。

四、AI 产物必须可被反驳

AI 产物最危险的地方，不只是它可能错，而是它很擅长把错误说得完整。

一个 AI 生成方案可能结构清晰、术语准确、覆盖了很多边界条件，看起来已经“考虑全面”。但如果它没有失败条件，就仍然只是一个漂亮候选。

真正要问的是：

  • 它解决什么真实问题？
  • 它不解决什么问题？
  • 哪些 case 会让它失败？
  • 哪些指标证明它没有价值？
  • 它增加的复杂度由什么收益证明？
  • 什么情况下它应该被删除、降级或停止使用？

如果这些问题答不上来，AI 产物就还不是工程资产。它只是“看起来像资产”的文本或代码。

AI 产物如果没有失败条件，就只是看起来像工程资产。

五、复杂度代谢不只是删除

说到“代谢”，人们很容易只想到删除。删除当然重要，但复杂度代谢不只是清理垃圾。

更准确地说，代谢包括四件事：

代谢动作含义工程对应
吸收真正进入系统，被理解、验证、维护code review、测试、owner、监控、文档
转化从一次性产出变成长期能力模板、规则、测试集、平台能力、团队规范
排出移除无法证明价值的复杂度删除无 owner 代码、合并重复实现、废弃过期链路
调节控制生成速度，避免超过吸收能力限制主链路接入、设置实验区、阶段性清理

删除只是代谢的一部分。更重要的是，系统要能决定什么可以变成自己的一部分，什么只能作为实验记录，什么应该及时排出。

代谢不是清理垃圾，而是决定什么能变成身体的一部分。

六、从“生成—使用”到“生成—约束—吸收—代谢—沉淀”

AI Coding 最危险的流程是：

生成 → 使用

这太短了。它把 AI 输出直接当成工程资产，跳过了判断、吸收和代谢。

更成熟的流程应该是：

生成AI 给出候选代码、方案或文档
约束明确业务边界、失败条件和硬约束
吸收review、测试、owner、监控和维护责任
代谢删除、合并、压缩、降级和停止
沉淀模板、规则、测试库、平台能力和共识

生成只是入口。吸收和沉淀才决定它是不是资产。

很多 AI 产物可以作为探索存在，但不能直接进入主链路；很多 prompt 可以作为经验记录，但不值得变成长期流程；很多代码可以作为 demo 存在，但不应该成为生产依赖。

生成只是入口，吸收和沉淀才决定它是不是资产。

七、每个 AI 产物都要有归宿

不是所有 AI 产物都要被消灭，但所有 AI 产物都要有归宿。

它可以被保留、被删除，也可以被降级。

保留条件

  • 有真实使用场景；
  • 有明确 owner；
  • 有测试或反例；
  • 有监控或可验证结果；
  • 有复用价值；
  • 有维护边界；
  • 有收益大于复杂度的证据。

删除条件

  • 无 owner；
  • 无调用；
  • 无测试；
  • 与已有能力重复；
  • 只服务一次性探索；
  • review 或维护成本高于收益；
  • 无法说明业务价值；
  • 复杂度超过收益。

降级条件

  • 暂时有启发，但不适合进入主链路；
  • 适合作为 demo，但不适合作为生产依赖；
  • 适合作为 prompt 片段，但不适合固化为流程；
  • 适合作为反例或测试素材，而不是长期模块；
  • 适合作为文档记录，而不是继续维护的代码。

这套归宿机制，比简单说“要删除代码”更重要。因为工程里最常见的不是立刻删除，而是大量东西处在模糊状态：好像还有点用，但没人负责；好像以后可能用，但没有证据；好像删除有风险，但保留也持续制造理解成本。

不是所有 AI 产物都要被消灭，但所有 AI 产物都要有归宿。

八、组织也要奖励代谢能力

AI Coding 的复杂度问题，不只是技术问题，也是组织激励问题。

组织通常更容易看见：

  • 谁写了代码；
  • 谁发了 PR；
  • 谁上线了功能；
  • 谁做了工具；
  • 谁跑了很多 AI 流程。

但组织不容易看见：

  • 谁避免了错误复杂度；
  • 谁删除了历史包袱；
  • 谁减少了 token 浪费；
  • 谁把复杂 prompt 固化成简单规则；
  • 谁砍掉了不值得继续的 Agent 链路；
  • 谁阻止了一个看似完整但风险很高的方案进入主链路。

如果组织只奖励产出痕迹，不奖励复杂度代谢，那么 AI 会放大这种偏差。团队会越来越会生成，也越来越难维护。

AI 时代更稀缺的工程能力，不是写得更多，而是知道什么不该写、什么不该留、什么不该进入主链路。

核心结论

  • AI Coding 的瓶颈正在从生成效率转向吸收能力。能写出来不等于能被系统接住。
  • 未被吸收的产出不是库存，而是负债。无 owner、无测试、无复用、无删除条件的 AI 产物会变成长期复杂度。
  • 代码量和 token 是成本，不是成果。它们可以作为观察指标，但不能作为价值目标。
  • AI 产物必须可被反驳。没有失败条件的方案，只是看起来像工程资产。
  • 复杂度代谢不只是删除。它包括吸收、转化、排出和调节。
  • 每个 AI 产物都要有归宿。保留、删除、降级，都比长期悬置更健康。
  • AI Coding 的核心治理问题，不是生产效率，而是复杂度代谢。
