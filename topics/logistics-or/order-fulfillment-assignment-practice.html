订单分仓 / 指派问题，是物流、供应链、电商履约系统里非常典型的一类运筹优化问题。它表面上是在回答“这个订单从哪个仓发”，本质上是在库存、时效、产能、成本、用户体验和业务策略之间做约束优化。

可以把它理解为：给定一批订单、多个仓库或履约节点、每个仓的库存和处理能力，决定每个订单、甚至订单中的每个商品由哪个仓履约，使整体成本、时效、库存健康度和履约成功率尽可能好。

一、问题本质：订单分仓 / 指派到底在解决什么？

1. 最直观的问题

用户下了一个订单：商品 A × 1，商品 B × 2，收货地址杭州，承诺时效次日达。系统里有多个仓：

仓库位置A 库存B 库存发杭州运费时效
上海仓上海10008 元次日达
杭州仓杭州10205 元当日/次日
武汉仓武汉10010012 元2 日达

这时系统要决定：整单从杭州仓发？A 从上海仓发、B 从杭州仓发？全部从武汉仓发？为了保护杭州仓库存，是否故意不用杭州仓？如果杭州仓快爆仓，是否切到上海仓？如果用户是高价值用户，是否优先保障最近仓和最快时效？这就是订单分仓 / 指派问题。

2. “分仓”和“指派”的区别

订单指派

订单指派通常指把一个订单整体分配给某个履约节点。例如：订单 O1 → 上海仓，订单 O2 → 杭州仓。特点是一个订单通常只产生一个包裹，用户体验好，工程实现简单，适合订单商品都能由同一个仓满足的场景；缺点是当库存分散时，履约成功率可能下降。

订单分仓

订单分仓通常指一个订单中的不同商品可以从不同仓发出。例如订单 O1 中商品 A 从上海仓发，商品 B 从杭州仓发。特点是可以提高库存利用率和履约成功率，但可能产生拆单、多包裹、多运费，并让用户体验、售后和物流跟踪更复杂。

类型决策粒度优点缺点
订单指派订单级简单、体验好、易解释库存分散时容易无解
订单分仓商品行级 / 数量级灵活、履约率高拆单成本高、体验差、系统复杂

3. 真实业务优化的不是“最近仓”

很多人刚开始会以为离用户最近的仓发货就好了。但真实业务不会这么简单。最近仓可能库存不足、产能不足、截单时间已过、某些 SKU 需要特殊包装，或者核心仓库存需要保护。

维度说明
库存仓里有没有货，库存是否足够，是否可售
运费哪个仓发货成本最低
时效是否能满足当日达、次日达、2 日达等承诺
产能仓库当天是否还有处理能力
拆单一单多包裹会增加成本和体验损耗
库存保护热门仓库存不能被低价值订单快速消耗
区域策略同区域优先，跨区加惩罚
商品限制某些商品只能从指定仓发，或不能混包
履约风险爆仓、天气、尾程不可达、服务商异常
用户优先级高价值用户、高时效订单优先保障
订单分仓 / 指派问题，是在库存、时效、产能、成本、体验和策略约束下，寻找一组可执行的履约决策。

二、从运筹角度看，这是什么问题？

1. 二分图匹配 / 指派问题

最简单版本是每个订单只能分给一个仓，每个订单-仓组合有一个成本，每个仓有产能限制，目标是选择总成本最低的分配方案。

xi,w = { 1，订单 i 分给仓 w；0，否则 }
min ∑i∑w costi,w · xi,w
∑w xi,w = 1
∑i xi,w ≤ capacityw

这是最基础的订单指派模型。

2. 带库存约束的指派问题

真实业务里，不能只看订单数量，还要看 SKU 库存。假设订单 i 需要商品 s 的数量为 demandi,s，仓 w 的商品 s 库存为 stockw,s。如果整单从一个仓发，需要保证该仓能满足订单中所有 SKU。

∑i demandi,s · xi,w ≤ stockw,s

这个模型已经比普通指派复杂很多，因为一个订单会同时占用多个 SKU 的库存。某个仓对订单是否可行，不再只取决于一条边的成本，而取决于多个商品库存是否同时满足。

3. 拆单分仓问题

如果一个订单可以拆成多个商品行，每个商品行可以分给不同仓，就变成更细粒度的问题。

yi,s,w = { 1，订单 i 中商品 s 由仓 w 发货；0，否则 }
∑w yi,s,w = 1
∑i demandi,s · yi,s,w ≤ stockw,s

如果商品数量也允许拆，比如商品 B 买了 3 件，可以 1 件从上海发、2 件从杭州发，那么变量可写成整数：

qi,s,w ∈ ℤ≥0

表示从仓 w 给订单 i 发商品 s 的数量。

4. 多目标优化问题

订单分仓不是单目标问题。常见目标包括：最小化履约成本、最小化超时风险、最小化拆单数量、最大化库存健康度、最大化订单履约率、最大化高优订单满足率、平衡各仓产能、减少跨区发货。

min α·运费 + β·拆单惩罚 + γ·超时惩罚 + δ·产能超限惩罚 + ε·库存保护惩罚

实际业务里，通常会把多个目标加权成一个目标函数，也可能把某些目标作为硬约束，例如 VIP 订单不能超时、库存不能超卖。

三、真实业务里会遇到哪些复杂性？

1. 库存不是一个静态数字

教科书里库存是“仓库 W 有 SKU S 库存 100”，但真实业务里会区分实物库存、可售库存、锁定库存、在途库存、安全库存、预占库存、异常库存等。因此分仓系统通常不能直接使用库存表中的数字，而要使用经过处理的可履约库存。

available_stock = real_stock − locked_stock − safety_stock
库存类型含义
实物库存仓里真实有多少
可售库存可以卖给用户的库存
锁定库存已经被订单占用但还没发货
在途库存补货路上，还没入仓
安全库存为特殊渠道或高优订单保留
预占库存分仓决策时临时锁定
异常库存破损、冻结、盘点差异

2. 时效不是简单的距离

近仓不一定最快，因为仓库当天截单时间可能已过，尾程配送资源可能不足，某些区域可能偏远不可达，仓库可能爆仓，快递服务商线路可能异常，商品也可能需要特殊包装。真实系统通常维护“仓-地区-服务”的时效能力矩阵。

仓库杭州主城区杭州郊区宁波上海
杭州仓当日达次日达次日达次日达
上海仓次日达次日达次日达当日达
武汉仓2 日达2 日达2 日达2 日达

3. 仓库产能是硬约束

如果只按成本最低分配，可能所有核心区域订单都压到一个核心仓，导致仓库爆仓。真实业务中产能可能按总单量、件数、冷链、大件、截单时间、人工排班等维度拆分。产能约束通常比成本权重更硬，因为超过产能会直接导致履约失败。

4. 拆单是重要业务惩罚

从库存角度看，拆单很好，因为更容易凑齐库存；但拆单会增加运费、包材成本、用户体验损耗、售后复杂度、物流轨迹复杂度和碳排放。因此系统一般会设置拆单惩罚。如果一个订单由多个仓发货，就加惩罚。

5. 热门仓库存要保护

核心仓离主要消费区域近、时效好、成本低。如果所有普通订单都优先使用核心仓，很快会出现高价值用户下单时核心仓没货，只能跨区发甚至超时。因此需要用库存机会成本来保护核心仓稀缺库存。

6. 订单是实时流入的

离线建模里，通常有一批订单，然后统一优化。但线上业务是每秒都有订单进来，每个订单需要几十到几百毫秒内给出分仓结果。这会导致全局最优和在线实时决策之间的矛盾。

离线全局优化在线实时决策
全局最优局部最优
可以慢必须快
需要完整订单池只能看到当前订单
适合大促预分配、仿真评估适合日常履约

真实系统通常会采用：离线计算策略参数、库存水位、仓优先级、成本矩阵；在线按规则和轻量优化快速决策；实时扣减或预占库存；异常时回滚或重算；后续通过仿真评估策略效果。

四、常见业务建模方式

模型 1：规则优先级模型

很多系统早期会先用规则实现：过滤无库存仓、不可达仓、无法满足时效仓、异常仓；再按同区域、成本、库存健康度、产能压力等规则排序。

candidate_warehouses = []
for warehouse in warehouses:
    if not has_stock(order, warehouse):
        continue
    if not can_deliver(order.address, warehouse):
        continue
    if not meet_promise_time(order, warehouse):
        continue
    candidate_warehouses.append(warehouse)

best = min(candidate_warehouses, key=lambda w: score(order, w))

优点：易解释、易上线、出问题方便排查、业务方容易理解。
缺点：局部最优、规则越来越复杂、不同规则之间容易冲突、很难平衡全局库存和产能。

模型 2：打分模型

给每个订单-仓组合计算一个分数，然后选择分数最低的仓。

scorei,w = 运费 + 拆单惩罚 + 时效惩罚 + 库存惩罚 + 产能惩罚 + 跨区惩罚
维度计算方式
运费真实物流报价
时效未满足承诺则加巨大惩罚，VIP 可直接判不可行
拆单每多一个包裹加固定成本
库存健康度库存越低、未来需求越高，惩罚越高
仓产能仓越接近满载，惩罚越高
区域跨区发货加惩罚

优点：比纯规则灵活，可以逐步调权重，易工程化，适合线上实时决策。
缺点：权重不好定，容易被局部 case 带偏，不保证全局最优，需要仿真平台评估。

模型 3：整数规划 / 混合整数规划

对于一批订单，可以建立全局优化模型。

min ∑i∑w costi,w · xi,w
∑w xi,w = 1
∑i demandi,s · xi,w ≤ stockw,s
∑i xi,w ≤ capacityw
xi,w ∈ {0,1}

优点：建模清晰，可以做全局优化，适合离线仿真、批量分仓、策略评估。
缺点：大规模时求解慢，线上实时使用困难，数据质量要求高，需要处理无解、超时和降级。

模型 4：启发式 + 局部搜索

真实业务中很常见：先用规则或贪心生成一个可行解，再做局部调整降低成本，例如交换两个订单的仓、调整部分订单到备用仓、减少拆单，并限定运行时间。

优点：比单纯贪心好，比整数规划快，工程上可控，可以做时间截断。
缺点：不保证全局最优，策略设计依赖经验，调试难度比规则高。

五、真实系统一般怎么落地？

典型订单分仓系统链路如下：

订单进入
  ↓
订单解析：地址、商品、数量、用户等级、承诺时效
  ↓
候选仓召回：哪些仓理论上可发
  ↓
约束过滤：库存、可达、时效、商品限制、仓库状态
  ↓
成本打分：运费、拆单成本、时效风险、库存机会成本、产能压力
  ↓
选择最优方案
  ↓
库存预占
  ↓
履约单生成
  ↓
失败回滚 / 降级 / 重新分仓

工程落地时最关键的不是“会不会写一个数学模型”，而是如何把约束、目标、策略、求解、回滚、监控、仿真、解释性结合起来，形成一个能在线上稳定工作的分仓决策系统。

业务约束

约束 1：整单优先

默认一个订单尽量由一个仓发出。如果一个仓可以满足整单库存和时效，则优先整单发货。业务含义是尽量减少包裹数量、售后复杂度和用户体验损耗。

约束 2：允许拆单，但有惩罚

如果一个订单被多个仓发货，每增加一个额外仓，惩罚 8 元。一个订单只由 1 个仓发，拆单惩罚为 0；由 2 个仓发，拆单惩罚为 8；由 3 个仓发，拆单惩罚为 16。

split_penaltyi = 8 · (used_warehouse_counti − 1)

约束 3：VIP 订单优先保障时效

VIP 订单不能违反承诺时效。普通订单如果无法满足承诺时效，可以延迟一级，但要加 20 元惩罚。普通次日达订单可以变成 2 日达，但 VIP 次日达订单不能变成 2 日达。

约束 4：核心仓库存保护

W1 是核心仓，库存要保护。如果普通用户订单使用 W1 中库存低于 3 的 SKU，需要额外加机会成本惩罚。库存 ≥ 5 不惩罚，库存 3~4 每件 +3，库存 1~2 每件 +8。VIP 订单不加该惩罚。

约束 5：仓库产能

每个仓有订单数产能和件数产能。如果一个订单拆成多个仓，每个仓都产生一个履约单，占用该仓 1 单产能；件数产能按该仓实际发出的商品件数计算。拆单不仅增加运费，也会增加多个仓的作业压力。

约束 6：库存不可超卖

每个仓每个 SKU 的总分配数量不能超过库存。

∑i qi,s,w ≤ stockw,s

约束 7：无解处理

真实系统不能因为某些订单无解就直接失败，需要定义无解处理策略。常见方式包括标记缺货、放宽普通订单时效、允许拆单、使用备用仓、等待在途库存、触发调拨、对部分商品取消履约、返回前端重新承诺时效。本练习中可以先要求所有订单必须履约；如果无解，再记录无解原因。

七、这个练习可以分成 4 个难度等级

Level 1：整单指派，禁止拆单

每个订单只能分配给一个仓。要求仓库必须有完整订单所需库存，必须满足订单承诺时效，不能超过仓库订单产能和件数产能，目标是总运费最低。

xi,w ∈ {0,1}
min ∑i∑w freighti,w · xi,w
∑w xi,w = 1
∑i demandi,s · xi,w ≤ stockw,s
∑i xi,w ≤ order_capacityw
∑i item_counti · xi,w ≤ item_capacityw

Level 2：整单指派 + 业务惩罚

在 Level 1 基础上，不只看运费，还要加入普通订单延迟惩罚、核心仓库存保护惩罚、跨区惩罚、产能紧张惩罚和高优订单体验约束。

min ∑i∑w (freighti,w + delay_penaltyi,w + inventory_protection_penaltyi,w + cross_region_penaltyi,w + capacity_pressure_penaltyi,w) · xi,w

这一层已经很接近真实线上打分系统。重点观察总运费是否下降、是否过度使用核心仓、普通订单是否被合理延迟、VIP 订单是否全部满足承诺，以及某些权重变化是否导致策略剧烈波动。

Level 3：允许拆单

决策变量 qi,s,w 表示订单 i 的 SKU s 从仓 w 发多少件。

qi,s,w ∈ ℤ≥0
∑w qi,s,w = demandi,s
∑i qi,s,w ≤ stockw,s

拆单惩罚需要引入 zi,w，表示订单 i 是否使用仓 w。

zi,w ∈ {0,1}
split_penaltyi = 8 · (∑w zi,w − 1)

拆单后运费通常要按履约单算：

∑i∑w freighti,w · zi,w

这一层的重点是理解：拆单提高可行性，但会带来履约单数量、运费、作业产能和用户体验方面的额外成本。

Level 4：加入线上工程视角

3. 实现打分函数

def score(order, warehouse):
    freight = get_freight(order, warehouse)
    delay_penalty = get_delay_penalty(order, warehouse)
    inventory_penalty = get_inventory_penalty(order, warehouse)
    capacity_penalty = get_capacity_penalty(order, warehouse)
    cross_region_penalty = get_cross_region_penalty(order, warehouse)
    return freight + delay_penalty + inventory_penalty + capacity_penalty + cross_region_penalty

打分函数最好输出明细，而不是只输出总分。否则后续很难解释为什么选某个仓。

4. 实现贪心分配

可以按 VIP 优先、当日达优先、SKU 越稀缺越优先、商品件数越多越优先、候选仓越少越优先的顺序排序。

for order in sorted_orders:
    candidates = get_candidate_warehouses(order)
    if not candidates:
        mark_unassigned(order)
        continue
    best = min(candidates, key=lambda w: score(order, w))
    assign(order, best)
    deduct_stock(order, best)
    deduct_capacity(order, best)

这一步很关键。运筹算法在真实业务里，不是只输出一个解，而是要能回答：为什么这么分？哪些约束导致成本上升？哪些订单无解？哪个 SKU 是瓶颈？如果增加核心仓某个 SKU 库存，会改善多少？如果某个仓产能下降 20%，影响多少？如果提高拆单惩罚，拆单率和成本怎么变化？这就是算法白盒化和归因分析的入口。

九、可以进一步扩展的方向

1. 从单订单决策扩展到订单池优化

先实现单订单贪心，再实现一批订单的全局优化。对比两者的总成本、拆单率、核心仓库存消耗、订单无解率和 VIP 满足率，可以直观看到“局部最优”和“全局优化”的差异。

2. 增加需求预测

核心仓库存保护不应该只看当前库存，还应该看未来需求预测。

protection_scorew,s = f(current_stock, forecast_demand, replenishment_lead_time)

未来需求越高、补货越慢、当前库存越低，保护惩罚越高。

3. 增加承运商选择

实际履约不只是“仓到地区”，还涉及承运商选择。可以把决策从“订单 → 仓”扩展成“订单 → 仓 → 承运商”，这会引入新的成本、时效、产能和线路约束。

4. 增加多温区 / 大件 / 特殊商品

例如冷链商品只能从冷链仓发，大件商品只能从大件仓发，易碎品不能和部分商品混包，生鲜商品必须满足更短时效。这些约束会让模型更接近真实业务。

5. 增加回滚和重算机制

真实线上系统里，分仓成功不代表最终履约成功。可能失败的环节包括库存预占失败、仓库拣货失败、承运商不可用、地址不可达、用户修改地址、订单取消或部分取消。因此系统需要支持分仓结果幂等、库存预占回滚、重新分仓、异常原因记录和下游状态同步。

十、总结

订单分仓 / 指派问题非常适合用来理解物流运筹，因为它同时包含数学建模、业务约束、在线实时决策、库存一致性、产能约束、用户体验、策略可解释性和离线仿真评估。

真正有价值的工作不是单纯调一个求解器，而是把复杂约束下的履约决策，做成一个稳定、可解释、可评估、可迭代的线上系统。
