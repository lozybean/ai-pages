匈牙利算法：从指派问题到行列变换

匈牙利算法是一种用于解决“一对一最优分配问题”的经典算法。它不贪心地选择局部最便宜的匹配，而是在整体上找到总成本最低或总收益最高的指派方案。

一、匈牙利算法解决什么问题？

匈牙利算法解决的是二分图中的最优匹配问题。更直白地说，它适合处理这样的问题：

有一批“人”和一批“任务”，每个人做每个任务都有一个成本或收益，如何做到“一人一个任务、一个任务一个人”，让总成本最低或总收益最高？

在物流和履约场景中，可以把左边的一组点理解成骑手、仓库、车辆、机器或工人；右边的一组点理解成订单、门店、任务、作业或职位。边上的数字可以是距离、耗时、费用、惩罚，也可以是收益、匹配得分或利润。

左侧对象右侧对象边上的数值目标
骑手订单配送距离 / 超时风险总配送成本最低
仓库订单履约成本 / 时效惩罚总履约成本最低
工人任务完成成本 / 耗时总作业成本最低
广告位广告预估收益总收益最高

如果是成本问题，直接求最小总成本；如果是收益问题，通常可以通过转换，把最大收益问题变成最小成本问题。

二、为什么不能只用贪心？

很多人第一反应会用贪心：每次选当前成本最低的一对。但指派问题的关键在于“互斥”：一个任务被某个人占用了，别人就不能再选它。局部最便宜的选择，可能堵死后面的好方案。

任务 A任务 B
人 112
人 21100

如果贪心先选“人 1 → 任务 A”，成本是 1，看起来很好；但剩下只能“人 2 → 任务 B”，成本 100，总成本是 101。

真正的最优解是：

分配成本
人 1 → 任务 B2
人 2 → 任务 A1
总成本3
指派问题不能只看单条边是否便宜，而要看一组匹配组合是否整体最优。匈牙利算法的价值就在这里。

三、匈牙利算法的核心原理

匈牙利算法的经典矩阵版本，可以先从三个动作理解：

1. 行列变换

每一行减去本行最小值，每一列减去本列最小值。这样会制造出很多 0，同时不改变最优解的结构。

2. 用 0 尝试匹配

0 表示在当前调整后的成本体系下很有潜力的匹配。算法尝试从 0 中选出互不冲突的一组完整匹配。

3. 覆盖与调整

如果 0 不足以形成完整匹配，就用最少的线覆盖所有 0，再对矩阵做调整，制造新的 0。

4. 得到最优解

当可以从 0 中选出每行一个、每列一个的不冲突匹配时，就找到了原问题的最优分配。

四、完整计算例子：3 人分 3 个任务

有 3 个工人 A、B、C，要分配给 3 个任务 1、2、3。每个人做每个任务的成本如下：

任务 1任务 2任务 3
工人 A413
工人 B205
工人 C322

目标是：每个工人分一个任务，每个任务只能被一个工人做，使总成本最低。

第一步：每一行减去本行最小值

每一行的最小值分别是：

行最小值
A1
B0
C2

逐行相减：

A 行：4, 1, 3 减 1 → 3, 0, 2
B 行：2, 0, 5 减 0 → 2, 0, 5
C 行：3, 2, 2 减 2 → 1, 0, 0
任务 1任务 2任务 3
A302
B205
C100

这一步的含义是：每个工人无论最终做哪个任务，都会先扣掉自己这一行的最低成本。因为每个方案都必须给每个工人分一个任务，所以这种统一扣减不会改变最优方案。

第二步：每一列减去本列最小值

当前矩阵中，各列最小值为：

列最小值
任务 11

任务 1 列减去 1，任务 2 和任务 3 列不变：

A202

C000

第三步：尝试用 0 完成一组不冲突匹配

现在 0 的位置是：A 只能选任务 2；B 只能选任务 2；C 可以选任务 1、2、3。

工人可选的 0 成本任务
A任务 2
B任务 2
C任务 1、任务 2、任务 3

A 和 B 都挤在任务 2 上，所以不能直接选出 3 个互不冲突的 0。算法需要继续调整矩阵。

第四步：用最少的线覆盖所有 0

当前所有 0 可以用两条线覆盖：一条线覆盖任务 2 列，一条线覆盖 C 行。

任务 1任务 2任务 3
A202
B105
C000

只用了 2 条线，小于矩阵规模 3，说明还没有形成完整匹配，需要继续调整。

第五步：找没有被线覆盖的最小值，并调整矩阵

没有被线覆盖的位置是 A1、A3、B1、B3，它们的值是 2、2、1、5，最小值是 1。

调整规则如下：

  1. 没有被任何线覆盖的元素，减去 1；
  2. 被两条线交叉覆盖的元素，加上 1；
  3. 只被一条线覆盖的元素，不变。
位置类型位置变化
未覆盖A1、A3、B1、B3减 1

单线覆盖A2、B2、C1、C3不变

C任务 3

它们互不冲突：每个工人一个任务，每个任务只被一个工人占用。

注意，最终成本要回到原始矩阵计算：

五、为什么行列变化不改变最优解？

这是理解匈牙利算法最关键的一点。

每一行减去一个固定值，本质上是：对某个工人来说，无论他最终被分到哪个任务，这个方案都会少算同样一笔钱。因为完整指派中每一行必须选一次，所以所有可行方案都会一起少算这些行最小值之和。

每一列减去一个固定值也一样。因为完整指派中每一列也必须被选一次，所以所有可行方案都会一起少算这些列最小值之和。

新方案成本 = 原方案成本 − 所有行扣减之和 − 所有列扣减之和

这个被减掉的总量对所有完整匹配方案都一样，所以不同方案之间的相对优劣不会改变。行列变换只是把成本矩阵平移到一个更容易观察 0 匹配的形式。

六、在业务中的适用边界

匈牙利算法适合处理标准的一对一最优匹配问题。它可以作为理解物流指派、订单匹配和运筹建模的基础工具，但不能直接覆盖所有真实业务复杂性。

适合不适合直接处理
骑手与订单一对一匹配一个骑手可接多个订单
工人与任务一对一分配任务有复杂前后顺序依赖
机器与作业一对一匹配机器有容量、班次、维护窗口
小批量仓库与订单的一轮简化指派订单有库存、时窗、拆单、路径和动态流入

真实物流系统里，如果涉及一对多、多容量、库存、多商品、车辆路径、动态订单流，通常需要进一步使用最小费用最大流、线性规划 / 整数规划、启发式算法、局部搜索、滚动优化或 OR-Tools 等方法。

在运筹学习中，匈牙利算法的意义不只是“会算一个矩阵”，而是帮助建立一个重要直觉：很多业务分配问题不能只靠局部贪心，而要把整体组合约束放进模型里。
