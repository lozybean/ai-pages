最小成本最大流算法：应用场景、建模原理与主要算法

最小成本最大流在最大流的容量约束之上，为每条边增加单位费用。它回答的问题不是单纯“最多能送多少”，而是“在尽可能多满足需求的前提下，怎样让总成本最低”。

一、它解决什么问题

最大流只关心从源点 s 到汇点 t 最多能送出多少流量。最小成本最大流在此基础上进一步考虑：每条边通过 1 单位流量要付出多少成本。

最小成本最大流的目标是：先让总流量最大；在所有最大流方案中，再选择总费用最小的一种。
最大化 flow 在 flow 最大的前提下，最小化 total_cost = sum(flow(e) * cost(e))

这类问题特别适合描述“资源分配 + 容量约束 + 成本优化”的场景。它比普通最大流多了费用目标，比普通最短路多了多单位流量和容量约束。

最大流视角

关注是否能满足尽可能多的需求，例如最多能发出多少货、完成多少任务、匹配多少订单。

费用流视角

在满足量已经尽可能大的前提下，进一步选择运输成本、等待成本、惩罚成本或机会成本最低的方案。

二、典型应用场景

只要一个业务问题同时出现“供给、需求、容量、单位成本、最大满足量”这些元素，就可以尝试把它转成最小成本最大流。

运输与调拨

物流

仓库、城市、分拨中心、门店之间有路线容量和运输单价。目标是尽可能满足需求，同时让总运费最低。

订单履约分仓

履约

订单可以由不同仓库履约，每个仓有库存和产能限制，不同“仓-订单”组合的运费、时效惩罚不同。

任务与人员分配

指派

人员、机器、车辆或服务节点可以执行多个任务，每个组合有不同成本。费用流可以自然支持一对多和容量上限。

排班与时段资源

调度

把时间段、班次、人力需求和加班成本建成网络，选择满足需求且总成本较低的排班方案。

网络路由与带宽分配

网络

多条路径有不同带宽容量和传输成本，需要在最大传输量下控制总体路由成本。

最大收益选择

收益

如果边权表示收益，可以把收益取负数，转化为最小成本问题；最小负成本就对应最大收益。

直觉判断句式：如果问题可以表述为“在尽可能满足需求的前提下，让总成本最低”，它大概率可以用最小成本最大流建模，或者至少可以作为一个很好的基线模型。

三、网络建模方式

最小成本最大流的输入是一个有向图。每条边同时带有容量和单位费用：

edge(u, v): capacity = cap(u, v) cost = cost(u, v) flow = f(u, v)

容量决定这条边最多能承载多少流，费用决定每通过 1 单位流要付出多少代价。

1. 常见建模骨架

源点
总供给入口 -> 供给侧
仓 / 人 / 车 / 机器 -> 决策边
可行关系 + 单位成本 -> 需求侧
订单 / 任务 / 门店 -> 汇点
总需求出口

2. 边的含义

边容量费用业务含义
s -> supply供给上限通常为 0仓库库存、人员可用次数、车辆可用载重。
supply -> demand可分配上限单位成本某个供给服务某个需求的运费、耗时、惩罚或负收益。
demand -> t需求量通常为 0订单、任务、门店或时段需要被满足的数量。

3. 最大流和指定流量的差别

有时业务要求“尽可能满足”，这就是最小成本最大流；有时业务要求“必须送出 K 单位流”，则是指定流量的最小费用流。

最小成本最大流

算法持续增广，直到无法再从源点到汇点送流。返回最大流量以及这个最大流量下的最低费用。

指定流量最小费用流

算法只需要送出目标流量 K。如果送不到 K，说明约束下需求不可完全满足。

四、残量网络与费用

最小成本最大流同样依赖残量网络。每条正向边都会配一条反向边：

正向边 u -> v: residual_capacity = cap - flow cost = c 反向边 v -> u: residual_capacity = flow cost = -c

反向边的费用为负数，表示撤销 1 单位原来的流量，可以把此前支付的费用退回来。

关键点

反向边不是现实业务中的反向运输，而是算法层面的“撤销与重排”。费用流的最优性正是依靠这种可反悔机制实现的。

为什么每次找最短费用增广路

在残量网络中，一条从 s 到 t 的路径代表一种可以继续增加总流量的调整方式。路径费用是这次增加 1 单位流量的边际成本。

path_cost = sum(cost(e)), e 属于增广路径

因此，最短费用增广路就是当前所有可增广方案中，边际成本最低的一种。沿它增加尽可能多的流量，再更新残量网络，重复这个过程，就能逐步构造低成本的最大流。

五、主要算法

最小成本最大流最常见的实现路线是“连续找最短费用增广路”。不同算法的差异主要在于如何在残量网络中找到这条最短路。

1. SPFA 最短路增广

SPFA 版本最容易理解，也最常作为入门实现。因为残量网络中会出现负费用反向边，所以不能直接用普通 Dijkstra。

while 存在 s 到 t 的最短费用路径: delta = 路径上的最小剩余容量 沿路径增广 delta total_flow += delta total_cost += delta * path_cost
特点说明
优点代码直观，能处理负边权，适合中小规模问题和学习推导。
缺点最坏复杂度不稳定，在大图或特殊数据上容易变慢。
适合教学、竞赛小数据、业务原型、需要快速验证建模正确性的场景。

2. Dijkstra + 势能优化

更稳定的工程实现通常使用势能，也叫 potential。它把每条边的费用调整成非负的 reduced cost，从而可以用 Dijkstra 快速找最短路。

reduced_cost(u, v) = cost(u, v) + potential[u] - potential[v]

势能调整不会改变路径之间的真实优劣关系，但可以让算法避开负边权带来的 Dijkstra 限制。

优势

在稀疏图上通常比 SPFA 稳定，适合规模更大的费用流问题。

成本

实现复杂度更高，需要正确维护初始势能、最短距离和每轮更新。

3. 原始对偶算法

Dijkstra + 势能优化通常也被称为原始对偶方法。它一边维护当前流这个原始解，一边维护势能这个对偶信息，通过最短增广路逐步满足最优性条件。

直观理解：原始对偶算法不是盲目找便宜路径，而是在一套价格体系下寻找仍然值得增广的路径，并持续修正这套价格体系。

4. 负环消除算法

另一类思路是先找到一个最大流，然后检查残量网络中是否存在负费用环。如果存在，就沿负环调整流量，降低总费用；直到没有负环，当前最大流就是最小费用最大流。

先求任意最大流 while 残量网络存在负费用环: 沿负环调整可行流量 降低 total_cost

这种方法理论直观，但工程中不如最短路增广常用，因为负环检测和调整的实现成本较高。

六、任务分配例子

假设有两个工人 A、B，两个任务 X、Y。每个工人最多做一个任务，每个任务需要一个人完成。成本如下：

工人任务 X任务 Y
A31
B24

1. 建图

s -> A, B: capacity = 1, cost = 0 A -> X: capacity = 1, cost = 3 A -> Y: capacity = 1, cost = 1 B -> X: capacity = 1, cost = 2 B -> Y: capacity = 1, cost = 4 X, Y -> t: capacity = 1, cost = 0

2. 最优结果

最优分配：A 做 Y，成本 1；B 做 X，成本 2。最大流量为 2，总成本为 3。

这个例子看起来可以人工算出来，但当工人、任务、容量、班次、时效惩罚和禁配关系同时存在时，费用流的建模优势会变得明显。

3. 为什么不是简单贪心

如果只按单条边从小到大选，可能会先占用某个任务，导致后续更高成本。费用流通过残量网络允许撤销已有匹配，从而在整体上重新组合。

建模提醒

费用流保证的是你建模后的线性目标最优。真实业务中的非线性成本、时间窗顺序、路径依赖和多目标偏好，可能需要扩展模型或改用整数规划、车辆路径算法、启发式搜索。

问题核心目标是否考虑费用适用范围
最大流流量最大否容量瓶颈、最大满足量、可行性上界。

可以把最小成本最大流理解成网络流建模中的通用升级版：最大流回答“能满足多少”，费用流进一步回答“用什么代价满足”。

八、建模检查清单

费用流的难点通常不在算法代码，而在把业务问题翻译成网络。建模时可以逐项检查：

节点
  • 哪些是供给方，哪些是需求方？
  • 是否需要增加中间层表达时间、区域、SKU 或状态？
  • 源点和汇点是否清晰？
边
  • 一条边代表什么可行关系？
  • 容量表示库存、产能、次数、载重还是需求量？
  • 费用是否是每单位流量的线性成本？
目标
  • 是尽可能多满足，还是必须满足固定流量？
  • 收益是否需要转成负成本？
  • 多目标是否已经转换成单一线性费用？
边界
  • 是否存在非线性成本或路径顺序依赖？
  • 是否需要整数变量之外的复杂业务规则？
  • 是否需要滚动优化而非一次性静态求解？
